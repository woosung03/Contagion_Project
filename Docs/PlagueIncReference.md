# Plague Inc. 나무위키 참고 자료 — 현재 구현 대비 보완점

원본 Plague Inc.(전염병 주식회사, 오리지널 게임)의 나무위키 문서 5개(시스템/전략/이벤트/상태)를 검토해
Contagion_Project(우리 클론)의 현재 구현과 비교했다. 이 문서는 **"원본 게임은 이렇게 하는데, 우리는 지금 이렇게
되어 있고, 이렇게 바꿀 수도 있다"**는 참고용 메모다. 실제 반영 여부는 다음 세션에서 결정.

참고 원본:
- https://namu.wiki/w/Plague%20Inc.%2F%EC%8B%9C%EC%8A%A4%ED%85%9C (시스템)
- https://namu.wiki/w/Plague%20Inc.%2F%EC%A0%84%EB%9E%B5 (전략)
- https://namu.wiki/w/Plague%20Inc.%2F%EC%9D%B4%EB%B2%A4%ED%8A%B8 (이벤트)
- https://namu.wiki/w/Plague%20Inc.%2F%EC%83%81%ED%83%9C (상태)
- (업적 문서는 본문이 너무 길어 가져오지 못함 — 필요하면 나중에 별도로 확인)

## 1. 국가별 상태 단계 (원본에 있으나 우리 프로젝트엔 없음)

원본은 국가별로 **사망자 비율**에 따라 6단계 상태를 가진다:

- 평시 상태 (사망자 거의 없음) — 치료제 연구 100% 가동
- 전면적인 붕괴 상태 (사망자 20% 이상) — 연구 가동률 그대로 유지되나 혼란 시작
- 무질서 팽배 (사망자 50% 이상) — 연구 가동률 50~70%로 하락
- 무정부상태 근접 (사망자 70% 이상) — 연구 가동률 20~30%로 하락
- 완전한 무정부 상태 (사망자 95% 이상) — 정부 몰락, 연구 완전 중단, 이후 감염자가 없어도
  치안 붕괴로 인한 추가 사망 발생
- 더 이상 존재하지 않음 (사망자 100%) — 국가 소멸

**우리 프로젝트 현재 상태**: `HumanResistanceManager.ApplyPolicy()`는 국가별 `healthFunding`을
개발수준(High/Mid/Low) + **전 세계 공통** `ResistanceStage`(plagueVisibility 기준 5단계)로만 결정한다.
즉 "국가 A는 이미 90% 죽어서 정부가 없는데 국가 B는 멀쩡하다" 같은 **국가별 개별 붕괴 상태**가 없고,
전 세계가 같은 단계를 공유한다. `Country` 클래스에도 자체 상태 필드가 없다.

**적용 제안 (다음 세션 백로그)**:
- `Country`에 `GetCollapseStage()` 같은 계산 프로퍼티 추가 — `deadCount / population` 비율로
  평시/붕괴/무질서/무정부근접/무정부/소멸 6단계 판정 (임계값 20/50/70/95/100%).
  `WorldState.IsPathogenEradicated`처럼 계산 프로퍼티로 두면 저장 데이터 늘리지 않고 구현 가능.
- `HumanResistanceManager.ApplyPolicy()`에서 `country.healthFunding`을 이 단계별로 추가 감쇠
  (예: 무질서 50~70%, 무정부근접 20~30%, 완전 무정부 0%) — 지금은 전역 단계만 반영해서
  일부 국가가 이미 궤멸했어도 자금 투자가 안 줄어드는 문제가 있음.
  이 기여도를 `SimulationManager.RunTick()`의 `cureIncrease` 합산에 그대로 흘려주면 됨(코드 변경 없이
  `healthFunding` 값만 바뀌므로 자연스럽게 연결됨).
  개발수준 UI 텍스트에도 국가 상태를 노출.
  개발수준별 감쇠 폭이 실제로 게임 후반 밸런스에 영향이 크므로 플레이테스트 필요.
  (설계 문서에 없는 새 규칙이라 CLAUDE.md의 "설계 문서 대비 결정" 섹션에도 반영 필요.)
- 완전 무정부 상태 국가는 감염자가 0이어도 치안붕괴로 소량 추가 사망 — `SimulationManager.RunTick()`에
  국가 상태가 완전 무정부일 때 `deadCount`에 소액 증가 로직 추가하면 재현 가능 (선택 사항, 우선순위 낮음).

## 2. 세계 전체 상태 단계 (원본과 우리 기준이 다름)

원본은 **감염률**과 **사망률**을 섞어서 6단계로 나눈다:
발견 안 됨 → 발견됨 → 빠르게 확산 중(감염 20%+) → 세계적임(감염 50%+) →
세계를 위협(사망 1~20%) → 인류 멸종 임박(사망 20%+).

**우리 프로젝트 현재 상태**: `WorldState.GetResistanceStage()`는 오직 `plagueVisibility`(질병 노출도, 감염/사망과
간접적으로만 연동) 하나의 값으로 5단계(NoAwareness/DiseaseReported/PublicHealthEmergency/NationalEmergency/
WorldCollapse)를 나눈다. 감염률이나 사망률을 직접 보진 않는다.

**적용 제안**: 지금 구조를 굳이 바꿀 필요는 없음 — `plagueVisibility` 자체가 감염률에 비례해서 증가하도록
`SimulationManager.RunTick()`에 이미 설계되어 있어(4.1 근처 코드) 결과적으로 비슷하게 동작함. 다만 UI
(`HudController`/`NewsFeedController`)에서 단계 이름을 표시할 때, 원본처럼 "사망자 수" 기준 문구
("인류 멸종 임박" 등)를 별도로 얹고 싶다면 `WorldState`에 사망률 기준 텍스트를 하나 더 추가하는 정도로 충분.
급하지 않음.

## 3. 난이도별 보정 (원본엔 있으나 우리는 치료 속도만 반영)

원본 난이도(Casual/Normal/Brutal/MegaBrutal)는 **치료 속도**뿐 아니라 **전염성 자체에도 보정**을 준다:
Casual은 전염성에 +보정(퍼지기 쉬움), Brutal부터는 전염성에 **-보정**(퍼지기 어려움, 그만큼 신중한 플레이
요구). MegaBrutal은 추가로 "감염자가 많아질수록 진화 비용이 기하급수적으로 증가"하는 페널티와
"심각성이 0이어도 일정 규모 이상 퍼지면 무조건 발각"되는 룰이 있다.

**우리 프로젝트 현재 상태**: `GameManager.GetDifficultyResearchMultiplier()`는 치료제 연구 속도만
Casual 0.7 / Normal 1.0 / Brutal 1.4 / MegaBrutal 1.8로 차등화한다. `SimulationManager`의
`globalSpreadFactor`(전염성 계수)는 난이도와 무관하게 고정값 하나(기본 0.15)를 모든 난이도에 동일 적용한다.
즉 지금은 "쉬움=치료 느림, 어려움=치료 빠름"만 있고 "쉬움=퍼지기 쉬움, 어려움=퍼지기 어려움"이 없다.

**적용 제안**:
- `GameManager`에 `GetDifficultySpreadMultiplier()` 추가 (예: Casual 1.3 / Normal 1.0 / Brutal 0.8 /
  MegaBrutal 0.6), `SimulationManager.RunTick()`의 `newInfected` 계산에 곱해주면 원본처럼 고난이도에서
  "일부러 신중하게 병을 숨기는" 플레이가 의미 있어짐. 지금은 난이도가 치료제 속도차만 주는 사실상
  "타이머 난이도"라 고난이도의 긴장감이 약할 수 있음.
  이건 코드 한두 줄 추가라 부담 크지 않음 — 다음에 원하면 바로 반영 가능.
  (업그레이드 트리 비용이 감염자 수에 비례해 기하급수적으로 오르는 MegaBrutal 특유 룰은 지금 우리
  `UpgradeNode`가 고정 비용 구조라 반영하려면 설계 변경이 필요 — 우선순위 낮음, 필요해지면 별도 논의.)

## 4. 이벤트 종류 확장 아이디어 (원본 이벤트 목록 중 우리 EventManager에 없는 것)

우리 `EventManager.cs`는 설계 문서 8절 기준 6개 이벤트(자연재해/정치불안/의료파업 = 병원체 유리,
WHO긴급회의/국제공조/백신임상성공 = 인류 유리)만 하드코딩되어 있다. 원본에는 이 외에도 게임에
직접 영향을 주는 이벤트가 많다. 그 중 우리 시뮬레이션 구조로 바로 옮길 수 있는 것들:

- **감염자 처형/해당지역 폭격**: 감염 초기 + 질병 노출도가 높을 때, 특정 국가의 감염자를 일부~전체
  즉시 제거하고 그 국가를 봉쇄. 우리 쪽엔 없는 "강한 역풍" 이벤트라 긴장감을 크게 높여줄 수 있음.
  `EventManager`에 `ApplyExecutionOrBombing()` 추가 — 대상국의 `infectedCount`를 확률적으로 대폭 삭감
  (예: 30~100%) + `isBorderClosed/isAirportOpen/isPortOpen`을 강제로 닫음. 특히 인구 적은 섬나라
  (그린란드류)에서 뜨면 사실상 그 나라는 재감염이 거의 불가능해지는 원본 특유의 긴장감 재현 가능.
- **자연재해로 인한 국가별 치료 자금 투자 한계치 감소**: 원본은 지진/쓰나미/홍수/테러로 사망자가 나면
  그 나라의 "치료 자금 투자 한계치" 자체가 낮아진다(우리로 치면 `healthFunding`의 상한선이 영구적으로
  깎이는 것). 우리 `Country`엔 이런 "상한선" 개념이 없어 healthFunding 자체가 유동적 — 재해 이벤트가
  뜨면 일시적으로 healthFunding을 깎는 정도로 단순화해서 넣을 수 있음.
- **첫 사망자 발생 이벤트**: 병이 발견된 후 첫 사망자가 나오면 치료제 개발 속도가 상승. 우리 쪽은
  `deadCount`가 0에서 올라가는 순간을 감지하는 로직이 없음 — `EventManager`에 1회성(oneShot) 이벤트로
  추가하기 쉬움(백신 임상 성공과 동일한 패턴).
- **올림픽/월드컵 같은 플레이버 이벤트**: 게임성에 큰 영향은 없고 텍스트 다양성용 — 우선순위 낮음.

이 항목들은 전부 `EventManager`의 기존 `NewsEventSettings` + `TryTrigger()` 패턴을 그대로 재사용할 수
있어서, 다음에 "이벤트 늘리자"고 하면 새 필드 몇 개 + 메서드 몇 개 추가로 가능함 (구조 변경 불필요).

## 5. 원본의 국경 폐쇄 우선순위 (사소하지만 참고용)

원본은 국가가 봉쇄될 때 항상 **공항 > 국경 > 항구** 순서로 닫는다 (한 번에 다 닫지 않고 순차적으로).
우리 `HumanResistanceManager.CloseBorders()`는 임계값을 넘으면 세 개(`isBorderClosed`, `isAirportOpen`,
`isPortOpen`)를 동시에 닫는다. 순차 폐쇄가 주는 긴장감(예: "항구는 아직 열려 있으니 배로 파고들 시간이
있다")을 살리고 싶으면 임계값을 3단계로 쪼개서 순서대로 닫는 방식으로 바꿀 수 있음 — 다만 지금 우리
게임엔 "국가 간 이동경로별 진화" 개념이 원본만큼 세분화되어 있지 않아서(감염경로 진화 트리 자체가
설계 문서 6절 업그레이드 카테고리 수준으로 단순화됨) 우선순위가 낮음.

## 요약 — 다음에 반영하면 좋을 순서

1. 국가별 사망률 기반 붕괴 단계 (1번) — `Country`/`HumanResistanceManager`만 건드리면 되고, 게임의
   "국가마다 다른 운명" 느낌을 가장 크게 살려줌.
2. 난이도별 전염성 보정 (3번) — 코드 몇 줄로 고난이도 긴장감 개선.
3. 처형/폭격 이벤트 (4번) — `EventManager` 패턴 그대로 재사용, 극적인 이벤트 하나 추가.
4. 나머지(세계 상태 텍스트, 자금 상한선, 국경 폐쇄 순서, 올림픽 등)는 급하지 않음.

원본에서 다루는 전염병 종류별(바이러스/곰팡이/기생충/프리온/좀비/뱀파이어 등)·전략 공략 문서는
우리 게임이 "국가 vs 병원체 시뮬레이션 + 인류 저항 AI" 구조라 원본의 유전자 코드/변이 시스템,
좀비·뱀파이어 같은 특수 병원체 콘텐츠와는 설계가 근본적으로 다르다. 그래서 전략 문서(namu.wiki 전략
페이지)의 세부 공략 순서는 참고 가치가 낮아 이 문서에서는 생략했다 — 필요하면 개별 요청 시 다시 확인.
