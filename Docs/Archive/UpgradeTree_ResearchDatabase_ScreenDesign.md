# UpgradeTree → Research Database 화면 설계 확정안

`UpgradeTree_ResearchDatabase_Investigation.md`에서 "노드+연결선 캔버스 → 연구 데이터베이스
리스트" 전환이 결정됐다. 이 문서는 그 결정을 실제 화면 설계로 구체화한다. **순수 UX/게임디자인
설계 문서 — 코드/UXML/USS 작성 없음.** `DESIGN.md`(토큰·컴포넌트 규정)와 `UI_Design.md` §7
(현재 트리 화면 설계, Step 62로 구현 완료)에 이미 있는 컴포넌트·명명 규칙을 그대로 재사용하고,
캔버스 레이아웃(절대좌표·연결선)에 관한 내용만 리스트 레이아웃으로 대체한다. 실제 UXML/USS/C#
구현은 다음 세션 이후 별도 작업.

**변경 범위 재확인** (`UpgradeTree_ResearchDatabase_Investigation.md` 0절과 동일 원칙): 데이터
계층(`UpgradeNode`/`UpgradeManager`/`DefaultUpgradeTreeFactory`, 카테고리/선행조건/DNA 비용/
해금 로직)은 전혀 손대지 않는다. 이 문서는 **오직 화면에 그리는 방식**만 다룬다.

---

## 1. 전체 화면 레이아웃

기존 `upgrade-header`(상단 크롬) → 콘텐츠 영역 → `detail-panel`(하단 상세) 3단 구조는 그대로
유지한다 — Step 62 구현이 이미 이 뼈대를 검증했고, 바뀌는 건 가운데 콘텐츠 영역이 "절대좌표
캔버스"에서 "세로 스크롤 리스트"로 바뀌는 것뿐이다.

```
┌──────────────────────────────────┐
│ [상단 크롬] DNA 표시 · 카테고리 탭 3개 · 광고 보너스 · 닫기  │ ← Surface Hierarchy 1단계
├──────────────────────────────────┤
│ [연구 목록 영역]                              │
│  ┌ 갈래 섹션 헤더 1 ─────────────┐            │
│  │  연구 항목 행 1                │            │
│  │  연구 항목 행 2                │  ← Surface │
│  │  ...                          │    2~3단계  │
│  └───────────────────────────────┘            │
│  ┌ 갈래 섹션 헤더 2 ─────────────┐            │
│  │  연구 항목 행 ...              │            │
│  └───────────────────────────────┘            │
│  (세로 스크롤, 카테고리당 15개 내외)              │
├──────────────────────────────────┤
│ [상세 패널] "연구 분석 콘솔" — tactical-panel   │ ← Surface Hierarchy 2단계
│  코드/명칭/효과/선행/비용/상태 data-row + CTA    │
└──────────────────────────────────┘
```

가장 큰 구조 변화는 **가로 방향 스크롤·확대축소가 완전히 사라진다는 것**이다. 지금은 카테고리당
9개 노드를 절대좌표로 배치하느라 열 간격을 140px까지 줄여야 했는데(1440px 기준 화면에서도
빠듯함), 리스트는 세로로만 쌓이므로 화면 폭 제약에서 자유롭다. 상세 패널이 화면 하단을 고정
점유하는 대신, 항목 선택 시 목록 영역 아래에 나타나는 방식(기존과 동일)을 유지할지, 별도
바텀시트로 뺄지는 §9에서 별도로 판단한다.

## 2. 탭 구조

기존 좌우 화살표(◀▶) 네비게이션을 **상단 고정 탭 3개**로 교체한다.

```
[ 전파 ]  증상    능력
  ▔▔▔▔
```

- **왜 화살표 대신 탭인가**: `UI_UX_Review.md` 10절이 이미 "지금 몇 번째 카테고리인지 진행
  표시가 없다"를 문제로 지목했고, `UI_Redesign_Plan.md` 6절은 이를 "1/3 인디케이터 추가"로
  처방했다. 탭 3개를 상시 노출하면 인디케이터를 별도로 만들 필요 없이 **탭 자체가 항상 위치를
  보여준다** — 처방이 아니라 원인 제거.
- 탭 개수가 3개로 고정이라(카테고리 enum 자체가 3종) 화면 폭 문제도 없다. 각 탭은 터치 영역
  48px 이상(`--touch-target-min`) 확보.
- 선택된 탭은 기존 선택 강조 규칙(`--color-border-selected` 금색 밑줄 또는 하단 발광 헤어라인)
  을 그대로 쓴다 — 새 색상 축을 만들지 않는다(DESIGN.md 색상 축 분리 원칙).
- 영문 LAB 캡션 병기 관행(§7.5 "TRANSMISSION LAB"/"SYMPTOM LAB"/"ADAPTATION LAB")은 탭
  라벨 자체보다는 목록 영역 헤더(섹션 상단)에 유지 — 탭은 짧은 한글("전파"/"증상"/"능력")로
  압축해 3개가 한 줄에 여유 있게 들어가게 한다.
- 기존 3-GameObject 구조(TransmissionTreeUI/SymptomTreeUI/AbilityTreeUI, 카테고리당 독립
  UIDocument)를 유지할지, 탭 전환을 위해 하나의 화면 안에서 리스트 콘텐츠만 갈아끼우는 구조로
  합칠지는 §10에서 구현 관점으로 별도 판단.

## 3. Branch(갈래) 표시 방식

**연결선 없이 "섹션 헤더 + 그 아래 세로 리스트"로 갈래를 표현한다.** 사용자가 제시한 예시
형식을 그대로 따르되, 실제 프로젝트 데이터(`NodeDisplayNames`, `FinalDecision.md`가 확정한
갈래 명칭)를 반영하면 아래와 같다. (※ 아래 노드→갈래 배정은 `NodeDisplayNames`와
`FinalDecision.md`에 명시된 갈래명·대표 노드 ID를 근거로 재구성한 예시다. 정확한 선행조건
그래프는 `DefaultUpgradeTreeFactory.cs`의 `position`/`prerequisites` 원본을 구현 시 재확인할 것.)

```
TRANSMISSION LAB

[공기 계열]
  공기 전파 I
  공기 전파 II
  비말 전파
  비말 전파 강화

[수인성 계열]
  수인성 전파 I
  수인성 전파 II
  곤충 매개 전파

[접촉·동물매개 계열]
  접촉 전파 I
  인수공통 전파
  인수공통 전파 강화
  혈액 전파
  혈액 전파 강화

[통합 연구] ← 위 3계열이 합류하는 후반 노드
  광역 전파 I
  광역 전파 II
  전지구적 전파
```

```
SYMPTOM LAB

[은신형 — 발진 계열]
  발진
  피부 병변
  피부염
  만성 병변

[표준형 — 기침 계열]
  기침
  발열
  폐렴
  호흡 부전

[공격형 — 구토 계열]
  구토감
  구토
  출혈
  패혈증

[통합 연구]
  다발성 장기부전 I
  다발성 장기부전 II
  전신 장기부전
```

설계 규칙:

- **섹션 헤더 = 갈래**, `tactical-caption` 톤(영문 대문자 or 강조 한글 + `tracking-caption`
  자간)으로 목록 항목과 시각적으로 분리한다. 색은 카테고리색(청/주황/보라, DESIGN.md §6)의
  좌측 accent-bar를 헤더에도 이어서 "이 섹션이 이 카테고리 소속"임을 계속 인지시킨다.
- **섹션 안 세로 순서 = 티어 순서.** 연결선이 하던 "이게 다음 단계다"라는 정보를, 순수하게
  "위에서 아래로 읽는 순서"로 대체한다 — 리스트 UI의 가장 기본적인 관용구라 학습 비용이 없다.
- **"통합 연구"(합류 노드) 섹션을 별도로 분리**한다. 두 계열 이상을 선행으로 요구하는 노드는
  단일 계열 섹션에 넣으면 "어느 계열 소속인지" 오해를 살 수 있으므로, 항상 목록 맨 아래
  독립 섹션으로 둔다 — 이 섹션의 첫 항목은 잠금 사유(§6)에 "선행: 공기 계열 또는 수인성 계열
  또는 접촉 계열 중 하나 완료" 식으로 다중 조건이 표시될 수 있음.
- 코너컷(`corner-cut`)은 섹션 헤더나 리스트 행에는 붙이지 않는다 — Usage Rules가 이미
  "수십 개가 반복되는 행에는 코너컷 대신 좌측 accent-bar"라고 못 박아 뒀다(DESIGN.md Corner
  Cut Don't). 코너컷은 하단 상세 패널(화면당 1개)에만 남는다.

## 4. 연구 항목 목록 구조

리스트 행 하나는 기존 `tree-node`의 4줄 스택(코드/이름/상태/비용)을 **가로 한 줄로 압축**한다
— 리스트 행은 카드보다 가로 폭이 넓으므로 4줄을 쌓을 필요가 없다.

```
[코드]  이름              [상태 칩] 상태라벨       DNA 비용/완료
TRANS-004  비말 전파 강화   ● 연구 가능           14 DNA
TRANS-005  광역 전파 I      ○ 잠김 · 선행: 비말 전파 강화   —
```

- 왼쪽부터: 좌측 4px accent-bar(카테고리색, 기존 tree-node 규칙 그대로) → 코드(`TRANS-004`,
  기존 `_codeByNodeId` 로직 그대로 재사용, §7.3 방식 그대로) → 이름(`DisplayName()` 그대로)
  → 상태 칩(`stat-chip`, 6×6px) + 상태 라벨 → 우측 정렬 비용/완료 텍스트.
  이는 기존 `data-row`의 "라벨:값" 2분할 구조를 3분할(코드+이름 / 상태 / 비용)로 확장한
  변형이며, 새 컴포넌트를 만들지 않고 `data-row` 레이아웃 계약(좌우 정렬, `grid-line` 하단
  구분선)을 그대로 따른다.
- **잠긴 행은 선행 사유를 같은 행 안에 인라인으로 붙인다**(위 예시 TRANS-005) — 별도 탭 없이
  바로 보인다. §6에서 상세히 다룬다.
- 행 높이는 `--touch-target-min`(48px) 이상을 보장한다. 잠긴 행에 선행 사유 문구가 길어져
  두 줄이 되면(예: "선행: 공기 계열 또는 수인성 계열 완료") 행 높이가 48px보다 커지는 것은
  허용 — 코드/이름/비용은 항상 1줄 고정, 상태+사유만 필요 시 2번째 줄로 줄바꿈.
- 완료(MAXED)된 행은 우측을 "완료"로 바꾸고(기존 로직 그대로), 배경을 브랜드 골드 계열
  옅은 톤(Surface Hierarchy 4단계, `surface-selected` 유사)으로 살짝 강조해 "이 계열은 여기가
  끝"이라는 인상을 준다 — 연결선이 주던 "종점" 신호의 대체.

## 5. 상태 표시 방식

`UI_Design.md` §7.2/§7.9가 이미 정의한 LOCKED/AVAILABLE/ACTIVE/MAXED 4단계 판정 로직과 색
규칙(브랜드 DNA초록=진행 중, 브랜드 골드=완료, 중립색=잠김/대기)을 **그대로 재사용**한다 —
캔버스든 리스트든 "노드 하나가 지금 어떤 상태인가"라는 게임 의미는 동일하기 때문이다.

| 상태 | 리스트 행에서의 표현 |
|---|---|
| LOCKED | 상태 칩 중립색(회백) + 라벨 "잠김" + 인라인 선행 사유(§6). 행 전체 opacity 살짝 낮춤(기존 `tree-node--locked` opacity 0.7 그대로) |
| AVAILABLE | 상태 칩 `accent-glow-soft` + 라벨 "연구 가능". DNA 부족 시(§6의 두 번째 사유) 비용 텍스트만 경고 톤으로 |
| ACTIVE | 상태 칩 브랜드 DNA초록 + 라벨 "진행 중"(=해금 완료, 더 진행할 다음 노드 있음) |
| MAXED | 상태 칩 브랜드 골드 + 라벨 "완료", 우측 텍스트 "완료"로 대체, 행 배경 옅은 골드 강조(§4) |

카테고리색(좌측 accent-bar)과 상태색(칩+라벨)이 같은 행 안에서 위치가 분리돼 있어(카드였을
때와 동일 원칙, DESIGN.md Node State Colors Do) 두 축이 섞이지 않는다. Severity 4색(감염/
사망/위험/정보)은 이 화면에 절대 쓰지 않는다 — 이미 확립된 축 분리 규칙 그대로.

## 6. 잠금 사유 표시 방식

`UI_UX_Review.md` B항목이 지목한 "잠긴 노드가 왜 잠겼는지 안 보임" 문제를 이번 전환에서
구조적으로 해결한다. 잠금 사유는 두 가지뿐이다 — 선행조건 미충족 / DNA 부족. 둘을 구분해서
표시한다.

- **선행조건 미충족**: "선행: [노드명]" 텍스트를 행의 상태 라벨 옆에 인라인으로 붙인다.
  선행이 여러 개면 쉼표로 나열(`, `), 3개 이상이면 "선행: 3개 항목"으로 축약하고 상세 패널
  탭 시 전체 목록을 보여준다(상세 패널은 이미 이 정보를 갖고 있음, §7).
- **선행은 충족했지만 DNA 부족**: 상태 라벨은 "연구 가능"(AVAILABLE)으로 유지하되, 우측 비용
  텍스트를 경고 톤(중립 회백이 아니라 살짝 어두운 톤 — severity 색과 겹치지 않는 별도 처리)
  으로 바꿔 "조건은 됐는데 자원이 없다"를 구분한다. `UI_Design.md` §7.2가 이미 "affordable
  여부는 available 안의 서브 톤"으로 분류해뒀던 것과 동일한 판단을 리스트 행에도 적용.
- 두 사유를 같은 자리(상태 라벨 옆)에 조건부로 표시하면 되므로, 새 UI 영역을 만들 필요가
  없다 — `stat-chip` + 텍스트 조합 안에서 문구만 바뀌는 정도.
- 이 정보는 **탭하지 않아도 리스트 단계에서 보인다**는 게 핵심이다 — 캔버스 카드는 폭이
  좁아 "상태: 잠김"만 반복했지만, 리스트 행은 가로 폭이 넓어 사유까지 한 줄에 들어간다.

## 7. 상세 패널 정보 구조

**변경 없음.** `UI_Design.md` §7.4가 설계하고 Step 62가 구현한 "연구 분석 콘솔"
(`tactical-panel` + `corner-cut` 4개 + `data-row` 반복)을 그대로 쓴다 — 이미 리스트형
구조(코드/명칭/효과별 행/선행 노드/DNA 비용/상태)라 트리든 리스트든 무관하게 동작하기
때문이다(`UpgradeTree_ResearchDatabase_Investigation.md` 9절에서 이미 확인).

유일한 추가 고려사항: 선행 노드가 여러 개인 합류 노드(§3 "통합 연구" 섹션)를 탭했을 때, 지금은
`AddDetailRow("선행 노드", string.Join(", ", ...))`로 전부 한 줄에 나열하는데, 조건이 "OR"인지
"AND"인지가 텍스트만으로는 구분되지 않을 수 있다 — 데이터 모델(`prerequisites: List<string>`)
자체가 AND/OR을 구분하지 않는 단순 리스트이므로, 이 화면 설계 범위에서는 표시 방식을 건드리지
않고 "필요 시 데이터 모델에 OR 조건 표현이 추가되면 그때 상세 패널 표시 방식도 갱신"으로
남겨둔다(현재 45개 노드가 실제로 OR 조건을 쓰는지는 `DefaultUpgradeTreeFactory.cs` 원본 확인
필요 — 이번 조사 범위 밖).

## 8. 연구 시작 CTA 구조

버튼 자체의 위치·활성화 로직(`CanUnlock()` 조회 → `SetEnabled()`, `TryUnlock()` 호출)은
변경 없음. 다만 **라벨을 "구매"에서 "연구 시작"류로 바꾸는 것을 권장**한다.

- "구매(Buy)"는 상점/RPG 아이템 구매 어휘다. Research Database 컨셉("병원체를 연구하고
  설계한다")과 어휘 차원에서 맞추려면 "연구 시작 (N DNA)" 또는 "연구 진행 (N DNA)"이 더
  정합적이다 — 사용자가 제시한 흐름도 자체가 "연구 시작"으로 끝난다.
- 상태별 텍스트도 함께 정리: `해금 전` → "연구 시작 (N DNA)", `이미 해금됨(ACTIVE/MAXED)` →
  기존 "이미 해금됨" 대신 "연구 완료"로 통일(상태 라벨과 어휘 일치).
- 버튼 등급은 `UI_Redesign_Plan.md` 2절이 이미 정의해둔 primary 버튼 등급(발광 테두리,
  `accent-glow`)을 적용 — 이는 트리든 리스트든 공통으로 처방된 사항이라 이번 문서에서
  다시 설계하지 않고 그대로 상속.
- 광고 보너스 버튼("광고 보고 DNA +10", 프리미엄 보라색)은 상단 크롬에 그대로 유지 — 목록
  구조 변경과 무관.

## 9. 모바일 세로 화면 UX

- **세로 스크롤 단일 축** — 가로 스크롤/확대축소가 완전히 사라져(1절) 1440×3120 세로 화면의
  기본 제스처(위아래 스와이프)만으로 전체 카테고리(15개 내외)를 탐색한다. 캔버스가 겪던
  "열 간격을 줄여야 화면에 들어간다"는 제약 자체가 사라진다.
- **상세 패널 위치**: 두 방향을 고려할 수 있다.
  (a) 기존처럼 화면 하단에 상시 고정된 패널로 두고 목록 영역을 그 위 남은 공간에서 스크롤 —
  구현 변경이 가장 적음(기존 UXML 구조 그대로).
  (b) 항목 탭 시 하단에서 올라오는 바텀시트로 전환 — 목록 영역을 화면 전체로 쓸 수 있어 한
  번에 더 많은 항목이 보이지만, 신규 애니메이션/오버레이 레이어가 필요해 구현 비용이 늘어남.
  **권장: (a) 유지** — 이미 검증된 구조고, 목록 항목 자체가 카드→행으로 압축되며 세로 공간이
  넉넉해져(4줄→1~2줄) 하단 고정 패널과의 공존 부담이 오히려 줄어든다.
- **섹션 헤더 스티키 여부**: 스크롤 중에도 "지금 어느 갈래를 보고 있는지" 헤더가 상단에
  붙어있게(sticky) 할지는 선택사항 — UI Toolkit `ScrollView`가 네이티브 sticky header를
  지원하지 않아(코드베이스 위키 확인 필요) 구현 비용이 있다면 생략 가능. 갈래당 항목이
  3~5개뿐이라 스크롤 한 번에 섹션 전체가 보이는 경우가 많아 sticky 없이도 방향 감각 손실은
  제한적이다 — **1단계에서는 생략, 실기기 테스트 후 필요성 재판단**.
- **터치 영역**: 리스트 행 48px 이상(§4), 탭 버튼 48px 이상(§2) — `--touch-target-min` 그대로
  적용. 카드보다 리스트 행이 오히려 오탭 방지에 유리하다(가로 폭 전체가 히트 영역이 되므로
  카드 좁은 폭보다 터치 성공률이 높음).

## 10. UI Toolkit 구현 친화성

(참고용 — 실제 구현은 하지 않음. "구현 시 무엇이 쉬워지는가"만 설계 근거로 남긴다.)

- **제거되는 것**: `RebuildTree()`의 절대좌표 계산(`_xOffset`, `canvasWidth/Height` 산출,
  `node.position.x/y` 기반 `style.left/top` 대입)과 `DrawConnections()`의 `Painter2D` 벡터
  계산(꺾은선 좌표, 포트 마커) 전체. `ScrollView.contentContainer`에 자식을 순서대로
  `Add()`만 하면 UI Toolkit의 기본 Flex 세로 흐름이 배치를 전부 대신 해준다 — 좌표 계산
  코드가 통째로 필요 없어진다.
  기존 `Redesign_Investigation.md`/`ResearchDatabase_Investigation.md`가 이미 이 부분을
  "8·9절, 순수 UI 레이어 교체"로 짚어뒀다.
- **그대로 남는 것**: `CreateNodeElement()`(→ 리스트 행 생성 함수로 개명되지만 내부 로직은
  `DetermineState()`/`GetEffectiveCost()`/`DisplayName()`/`_codeByNodeId` 조회 그대로),
  `SelectNode()`/`AddDetailRow()`(상세 패널), `NodeDisplayNames`/`EffectStatLabel()`/
  `CategoryLabel()` 전부 무변경.
- **새로 필요한 것은 최소 수준**: 갈래 섹션 헤더를 만드는 헬퍼 함수 하나(갈래별로 노드를
  묶어 헤더+리스트를 순서대로 Add) 정도 — 이미 `RebuildTree()`가 `nodes.OrderBy(...)`로
  정렬하고 있으므로, 정렬 기준에 "갈래"만 한 단계 더 넣으면 된다.
- **UI Toolkit `ListView` 컴포넌트는 도입하지 않는 것을 권장**한다 — `ListView`는 수백~수천
  행의 가상화(virtualization)가 필요할 때 쓰는 컴포넌트인데, 카테고리당 15개 내외의 소규모
  리스트라 가상화 이점이 없고, 이 프로젝트의 다른 리스트 화면들(Country Dock, 국가 48행
  리스트, CountryStatusPanel)도 전부 `ListView`가 아니라 "컨테이너 하나 + C# 루프로 행
  Add()" 패턴을 쓴다 — 새 컴포넌트를 도입하기보다 이미 검증된 패턴을 그대로 따르는 편이
  일관성·리스크 양쪽에서 유리하다.
- **종합**: 코드량 기준으로는 순감소(캔버스 좌표·연결선 계산 제거분이 리스트 정렬 헬퍼
  추가분보다 큼), 재사용률은 매우 높음(상태 판정·비용·표시명·상세 패널 로직 100% 유지) —
  `UpgradeTree_ResearchDatabase_Investigation.md` 8·9절의 평가와 일치한다.
