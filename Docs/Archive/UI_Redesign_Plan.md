# UI_Redesign_Plan.md — Contagion Project UI 리디자인 계획서
## "Tactical Command Center" 컨셉 강화

> **순수 설계 문서. 코드 작성 금지, UXML 수정 금지.** `Docs/UI_UX_Review.md`(스크린샷 4장 기반
> 진단)의 A/B/C 분류를 근거로, 실제 착수 시 무엇을 어떻게 바꿀지에 대한 **설계안**만 정리한다.
> 색상·타이포·컴포넌트 규칙의 정본은 여전히 `Docs/DESIGN.md`이며, 이 문서가 제안하는 신규

## 0. 이 계획서가 푸는 문제 — 한 줄 요약

`UI_UX_Review.md`의 결론은 "패널·데이터 표시는 전술 언어로 전환됐지만, 화면 점유율이 가장 큰
**지도**와 상호작용 빈도가 가장 높은 **버튼**은 전환 대상에서 빠져 있다"였다. 이 계획서는 그
두 영역을 포함해 리뷰의 A/B 항목 전체를 화면별 실행 가능한 설계안으로 옮긴다.

## 1. 세계지도 시각 스타일 개편안

**문제**
GamePlay 화면 면적의 절반 이상을 차지하는 세계지도(`Resources/WorldMap/world_base.png`,
4000×1714px 정적 텍스처)가 밝은 청록 바다·흰 대륙 톤으로 제작돼 있어, DESIGN.md Design
Principle #2("Always-dark canvas" — 어떤 화면도 밝은 캔버스로 전환되지 않음)와 화면에서 가장
큰 면적으로 충돌한다. 교통 유닛 아이콘(`plane.png`, `Ship 1.png`)도 원래 이모지(Twemoji)에서
파생된 캐주얼한 룩을 그대로 쓰고 있어, DEFCON/Frostpunk류 군사 HUD 레퍼런스와 톤이 어긋난다.

**목표**
지도를 "다크 캔버스 위에 발광선/신호로 표시되는 레이더 스코프"로 재해석해, HUD 크롬(상하단
바)과 지도가 같은 시각 언어를 공유하게 한다. 국가 상태(healthy/infected/dead)와 교통망은 이미
severity 축·accent-glow 축을 쓰고 있으므로, 그 신호가 다크 배경 위에서 지금보다 더 도드라지게
만드는 것이 핵심이다.

**해결 방안**
2단계로 나눠 제안한다.

- **완화책(저비용, 우선 적용 가능)**: `world_base.png` 원본은 그대로 두고, 지도 레이어 위에
  반투명 다크 스크림(예: `--color-bg-scrim-soft`/`--color-bg-scrim-strong` 계열, 알파값은
  실기기 검증 후 확정)을 얹어 전체 밝기·채도를 낮춘다. 지도가 완전히 안 보일 정도로 어둡게
  하지 않고, "다크 필터를 통해 보는 지도"처럼 톤만 낮추는 것이 목표다. 코드 변경 없이 지도
  렌더링 레이어 순서 위에 오버레이 하나를 추가하는 수준이라 가장 빠르게 시도해볼 수 있다.
- **근본책(고비용, 중장기)**: `world_base.png` 원본을 다크 팔레트(짙은 남색/차콜 바다,
  저채도 어두운 대륙, 옅은 격자선으로 경위선 표현)로 재도색한다. 이때 국가별 오버레이
  실루엣(`Resources/CountryShapes/{countryId}.png`, 48장)과 감염 점 오버레이 색상도 새
  배경 대비로 재검증이 필요하다 — 배경이 어두워지면 감염 점(주황/빨강)의 대비가 오히려
  강해지는 효과가 기대되지만, 실제로는 48개국 전수 재확인이 필요한 작업이다.
- **교통 유닛 아이콘**: 로더 코드(`TransportUnit.cs`)는 그대로 두고 스프라이트 애셋만
  교체한다 — 캐주얼한 실사풍 비행기/배 그림 대신, 단색 실루엣 아이콘(accent-glow 또는
  grid-line 색조의 심플한 선화)으로 바꾸면 "레이더에 잡힌 신호"처럼 보이게 할 수 있다.
  DEFCON의 항공기 삼각형 아이콘과 유사한 방향.
- **그리드라인 오버레이(선택)**: 지도 위에 옅은 경위선(`--color-grid-line`, 매우 낮은 알파)을
  추가하면 UpgradeTree 노드 캔버스에 이미 쓰인 "회로 기판 재질감" 문법을 지도에도 재사용할 수
  있어 화면 간 일관성이 강화된다.

**기대 효과**
화면에서 가장 넓은 영역이 다크 캔버스 원칙에 합류하면서, HUD 크롬만 전술적이고 정작 게임의
주 무대(지도)는 캐주얼하다는 지금의 "이중 언어" 문제가 근본적으로 해소된다. 감염 확산이
어두운 배경 위에서 더 뚜렷한 "경보 신호"처럼 읽혀 게임의 긴장감 전달에도 기여한다.

## 2. 버튼 시스템 재설계안

**문제**
HUD의 `tab-button`/`tab-button--secondary`(Hud.uss, 테두리형·무배경)만 DESIGN.md의 각짐+
헤어라인 규칙을 따르고, 나머지 화면의 버튼 — MainMenu(`mainmenu-next`/`mainmenu-back`),
CountrySelect(뒤로/시작), UpgradeTree(`detail-panel__buy`), EndingScreen
(`ending-button`/`ending-button--revive`), RankingPanel(`ranking-button`) 등 — 은 전부
단색 배경을 꽉 채운 둥근 사각형이다. 버튼 클래스는 6개 USS 파일에 흩어져 있고 공용
"buttons.uss" 같은 단일 정본이 없다. 결과적으로 패널은 각졌는데 정작 플레이어가 가장 자주
누르는 요소만 화면마다 다른 톤의 "일반 모바일 버튼"으로 남아 있다.

**목표**
HUD의 `tab-button` 문법(배경색이 아니라 테두리로 위계를 표현)을 시스템 표준으로 승격시켜,
모든 화면의 주요 액션 버튼이 같은 규칙을 공유하게 한다.

**해결 방안**
- **버튼 등급 3종 정의(개념 설계)**: 주 액션(primary — accent-glow 1~2px 테두리 + 옅은
  배경, 예: 시작/다음/구매), 보조 액션(secondary — 중립 `--color-border` 테두리, 예:
  뒤로/취소), 프리미엄 액션(premium — brand-premium 보라 테두리, 예: 광고 보기). 3등급 모두
  배경은 채우지 않거나 아주 옅은 톤(surface 축)만 쓰고, 테두리 굵기·색으로만 구분한다 — 이미
  DESIGN.md Interaction Rules가 정의한 "`:active` 시 scale만 적용, 배경색은 건드리지 않는다"
  규칙과도 자연스럽게 맞물린다.
- **코너컷 대신 미니 브라켓**: 코너컷은 화면당 개수가 적은 패널 전용 문법이라 반복되는
  버튼에는 부적합(DESIGN.md Usage Rules와도 일치). 대신 버튼 좌우 끝에 짧은 대괄호(`[  ]`
  형태의 텍스트 프리픽스/서픽스 또는 얇은 세로 바 2개)로 "탭 가능한 콘솔 스위치"라는 인상만
  가볍게 준다 — 코너컷 문법을 억지로 재사용하지 않고 버튼 전용의 절제된 신호를 별도로 정의.
- **단계적 마이그레이션 원칙**: 기존 클래스는 제거하지 않고 신규 공용 버튼 클래스를 병기하는
  프로젝트 기존 원칙(UI_Design.md §1)을 그대로 따른다 — 화면별로 한 번에 하나씩 전환 가능해
  리스크가 낮다.
- **적용 순서 제안**: MainMenu/CountrySelect(다음/시작/뒤로) → UpgradeTree(구매) →
  EndingScreen/RankingPanel 순 — 근거는 7절 우선순위 참고.
- **비활성(disabled) 대비 보강**: 현재 UpgradeTree "구매" 버튼처럼 비활성 시 거의 배경과
  같은 무채색이 되는 문제를, primary 버튼 등급에서 "테두리는 남기고 텍스트만 흐리게" 하는
  규칙으로 보완 — 비활성이어도 "여기 버튼이 있다"는 존재감은 유지.

**기대 효과**
플레이어가 실제로 손가락을 대는 지점에서 정체성이 끊기지 않아, 화면 전환 시마다 "전술
콘솔 → 캐주얼 앱 → 전술 콘솔"로 톤이 오갔던 문제가 사라진다. 향후 신규 화면(설정/알림 등)도
버튼 컴포넌트가 이미 표준화돼 있어 처음부터 일관된 톤으로 시작할 수 있다.

## 3. 병원체 선택 화면 개선안

**문제**
`pathogen-card`는 이미 `tactical-panel`+코너컷 4개가 적용돼 있으나, 실제로는 선택된 카드의
굵은 금색 테두리만 두드러지고 미선택 카드의 코너컷·헤어라인은 존재감이 약하다. 또한 선택
시 나타나는 상세 패널이 리스트 카드와 같은 병원체 이름을 다시 헤더로 보여줘, 좁은 세로
화면에서 정보가 중복된다.

**목표**
"이 화면 전체가 판독 콘솔"이라는 인상을 선택 여부와 무관하게 모든 카드에서 느끼게 하고,
리스트(요약)와 상세 패널(심화)의 역할을 명확히 나눠 이름 중복을 없앤다.

**해결 방안**
- **미선택 카드 대비 강화**: 코너컷 색상·테두리 알파를 지금보다 한 단계 올려(단, 선택
  카드의 금색과는 명확히 구분되는 밝기로) 미선택 상태에서도 "판독 카드"라는 인상이 살도록
  한다. 선택 강조(금색)와 판독 프레임(발광 초록)이 서로 다른 축이라는 DESIGN.md 원칙은
  유지하면서 후자의 존재감만 보강하는 방향.
- **상세 패널 헤더 재구성**: 병원체 이름을 상세 패널 헤더에서 제거하고, 대신 "SPECIMEN
  ANALYSIS" 같은 영문 캡션 + 유형 아이콘(세균/바이러스/곰팡이 구분 심볼)만 남긴다 — 이름은
  리스트 카드에서 이미 충분히 확인했다는 전제.
- **대안(택1 검토)**: 상세 패널을 별도 영역으로 분리하는 대신, 선택된 리스트 카드 자체를
  아코디언처럼 확장해 그 안에 FlavorText를 펼치는 구조도 검토 가치가 있다 — 이 경우 이름
  중복 문제는 구조적으로 사라지지만, 기존 "리스트+고정 상세패널" 레이아웃(§8 기존 설계)을
  더 크게 바꾸는 것이라 리스크가 크다. 이번 계획서는 헤더 재구성(첫 안)을 우선 권장하고
  아코디언 안은 후순위 검토 항목으로만 남긴다.
- **전체 병원체 수 확인**: 스크린샷상 3종만 노출되고 있어(설계상 6종), 나머지 3종이
  스크롤로 접근되는지 카드 높이 조정이 필요한지 실기기 확인 후 카드 밀도를 재조정한다.

**기대 효과**
게임 진입 첫 화면부터 "판독 콘솔" 톤이 일관되게 느껴져 첫인상이 개선되고, 상세 패널의
역할(이름 재확인이 아니라 추가 정보 제공)이 명확해진다.

## 4. 국가 선택 화면 개선안

**문제**
48개국 리스트에서 국기 원색이 개발수준 기반 accent-bar(severity 색상 신호)보다 시각적으로
더 크게 작용해, "전략적으로 유리한 발원지 선택"이라는 목적보다 "익숙한 국기"라는 감정적
신호가 앞설 위험이 있다. 또한 상세 패널이 화면 하단을 고정 점유해 리스트 가시 영역이
5~6개국 수준으로 좁다.

**목표**
국기는 식별용 보조 정보로 톤을 낮추고, 의료수준·인구 같은 전략 신호가 먼저 읽히게 한다.
동시에 48개국 스캔 효율을 높인다.

**해결 방안**
- **국기 시각적 비중 축소**: 국기 아이콘 크기를 줄이고 채도를 낮추거나(예: 옅은 다크 프레임으로
  감싸 원색 노출 면적 축소) accent-bar 두께·채도를 상대적으로 키워, 국기보다 accent-bar가
  먼저 인지되도록 우선순위를 뒤집는다. 국기를 완전히 제거하지 않는 이유는 "국가를 식별한다"는
  기능 자체는 유효하기 때문 — 비중만 조정한다.
- **상세 패널 축소/접기**: 상세 패널을 상시 고정 노출 대신, 국가를 선택했을 때만 하단
  시트(bottom sheet) 형태로 확장되고 평소엔 접힌 한 줄(선택된 국가명 + 시작 버튼)만
  차지하도록 바꾼다 — Country Dock에 제안하는 collapsed/expanded 2단계 패턴(5절)과 동일한
  철학을 공유해 시스템 전체의 일관성도 높아진다.
- **정렬/필터 옵션(제안, 범위 확장)**: 인구순/의료수준순 정렬 토글을 헤더에 추가하면 48개국
  탐색이 "이름순 스크롤"에서 "전략 기준 스캔"으로 바뀐다 — 이는 기존 UI_Design.md 범위를
  넘어서는 제안이라 별도 검토가 필요한 항목으로 표시해둔다.

**기대 효과**
발원 국가 선택이 감정적 선택에서 전략적 선택으로 무게 중심이 이동하고, 리스트 가시 영역
확보로 48개국 탐색 피로가 줄어든다.

> 목록, 커밋 2e4483e) — 상시 노출되는 "독" 형태 자체가 사라져 "collapsed/expanded 2단계화"
> 제안은 전제가 없어져 **더 이상 적용 대상이 아니다.** 다만 이 절의 나머지 제안(비율/절대값
> 정보 위계 분리, 하단 터치 세이프존 확보)은 population-bar/graph-panel/action-strip이
> 여전히 존재하므로 Country Dock 제거와 무관하게 유효하다.

**문제**
Country Dock이 선택 전에도 인구/감염률/사망률/의료수준/국경/상태/공항·항구 7행을 전부
"-" 값으로 표시해 화면의 상당 부분을 빈 정보로 점유한다. `world-status-frame`이 이미 갖고
있는 "값이 있을 때만 크롬 등장" 철학과 다른 컴포넌트라 시스템 안에서 예외적으로 동작한다.
또한 population-bar(비율)와 하단 스탯 3분할(절대값)이 근접한 위치에서 같은 정보를
두 형태로 중복 노출한다.

**목표**
Country Dock을 world-status-frame과 같은 철학으로 통일하고, 중복 정보의 노출 위계를
재조정해 화면 밀도 대비 유효 정보 비율을 높인다.

**해결 방안**
- **Country Dock 2단계화**: 평시(미선택) 상태는 "국가를 탭하세요" 한 줄 + 작은 아이콘만
  차지하는 collapsed 형태로 축소하고, 국가를 탭했을 때만 7행 data-row가 펼쳐지는 expanded
  형태로 전환한다 — world-status-frame이 이미 검증한 "평시엔 라벨 폭이 0에 가깝게 줄고 위험
  상태일 때만 강조"라는 동일 패턴을 재사용하는 것이라 새 컴포넌트를 만들 필요가 없다.
- **비율/절대값 정보 위계 분리**: population-bar(비율)는 상시 노출 유지, 하단 스탯 3분할
  (절대값)은 그래프 패널을 탭했을 때만 펼쳐지는 형태로 바꾸거나, 반대로 절대값을 상시 노출
  하고 비율 바는 국가현황 패널로 이동하는 두 방향 중 하나를 택일한다 — 어느 쪽이든 "같은
  정보를 두 형태로 동시에 상시 노출"하는 현재 상태만 해소하면 된다.
- **레이어 대비 강화**: event-dock/country-dock(Surface Hierarchy 2단계, Docked panel)과
  population-bar/graph-panel(1단계, Scrim bar)의 배경 알파 차이를 지금보다 크게 벌려, 어느
  것이 "떠 있는 패널"이고 어느 것이 "화면에 고정된 크롬 바"인지 시각적으로 더 뚜렷하게
  구분한다.
- **하단 터치 세이프존 확보**: 업그레이드/국가현황/랭킹 버튼과 바로 위 그래프·스탯 텍스트
  사이 여백을 표준 spacing 토큰(`--space-md` 이상)으로 확보해 실기기 터치 시 오탭 위험을
  낮춘다 — 실측 검증은 QA 영역이지만 여백 자체는 이 리디자인 단계에서 미리 확보해둘 수 있다.

**기대 효과**
게임을 막 시작한 플레이어가 처음 보는 화면 우상단이 "정보 없음"으로 채워지는 문제가
사라지고, 화면 전체의 정보 위계(무엇이 항상 보여야 하고 무엇이 필요할 때만 보여야 하는지)가
명확해진다.

## 6. 업그레이드 시스템 UX 재설계안

**문제**
잠긴 노드는 전부 동일한 "상태: 잠김" 텍스트만 보여줘 무엇이 부족한지(선행 미해금/DNA 부족)
카드에서 바로 알 수 없다. 비용(DNA 수치)이 카드에 상시 노출되는지 상세 패널에서만 보이는지
일관되지 않은 것으로 보이며, 카테고리 전환(◀▶) 시 "지금 몇 번째 카테고리인지" 진행 표시가
없다. 더 근본적으로는 당시 별도 조사가 이미 진단했듯(조사 문서는 결론 흡수 후 삭제됨) 45개
노드 중 71%가 단일 효과라 "정답 트리"가 존재하는 구조라, UI를 아무리 다듬어도 판독할 전략적
긴장 자체가 부족하다. **[문서 무결성 감사, 2026-07-22]** 이 진단은 여전히 유효하다 —
`UpgradeTree_FinalDecision.md`(수치 재설계)는 구현 완료됐지만 그건 비용/효과 수치 조정 수준이고,
여기서 지적하는 "단일 효과·카테고리 간 무상호작용" 문제를 실제로 해소하려면
`ResearchDatabase_RuntimeSystems.md`/`UpgradeTree_ResearchDatabase_NodeMapping.md`가 다루는
질적 효과 다양화(Mechanic Unlock 등, Phase 2+)가 필요한데 이건 아직 착수 전이다(코드 확인:
`medicalBurdenModifier`/`unlockedFlags` 등 미구현).

**목표**
UI 레벨에서는 노드 카드 자체가 "왜 안 되는지/사면 무엇이 바뀌는지"를 탭 없이 전달하게 하고,
데이터 레벨에서는 이미 존재하는 조사 문서의 MVP안을 UI 설계와 정합시킨다.

**해결 방안**

*UI 레벨(이 계획서 범위)*
- **잠금 사유 표시**: `stat-chip`(6×6px 색상 칩) 컴포넌트를 재사용해 잠긴 카드에 "선행 필요"
  (중립색)와 "DNA 부족"(경고색 계열, 단 severity 4색과는 구분되는 톤)을 구분 표시한다 —
  신규 컴포넌트를 만들지 않고 기존 stat-chip 문법 확장만으로 해결 가능.
- **비용 상시 노출 통일**: 모든 노드 카드가 잠김/해금 가능 상태와 무관하게 비용(또는 "완료")
  텍스트를 항상 같은 위치에 표시하도록 통일한다 — §7.3 설계(코드/이름/상태/비용 4줄 스택)를
  전 노드에 예외 없이 적용.
- **카테고리 진행 표시**: 헤더에 "1/3" 같은 짧은 인디케이터를 추가해 3개 카테고리(전파/증상/
  능력) 중 어디를 보고 있는지 항상 알 수 있게 한다.
- **구매 버튼 등급 승격**: 2절에서 정의한 primary 버튼 등급을 적용해 지금의 존재감 약한 회색
  버튼 문제를 해소한다.

*데이터 레벨(별도 문서에 위임, 이 계획서에서는 UI 설계와의 정합성만 확인)*
- **[문서 무결성 감사, 2026-07-22] 상태 갱신**: 비용/효과 수치 재설계(구 6절 1단계 MVP —
  비용 곡선 비대칭화, 표시 라벨 재정의)는 `UpgradeTree_FinalDecision.md` 기준으로 **구현
  완료됐다**(코드 확인: `DefaultUpgradeTreeFactory.cs` 수치 일치). 다만 그보다 더 근본적인
  "노드 효과 다중화"(Mechanic Unlock, 카테고리 간 상호작용)는 `ResearchDatabase_RuntimeSystems.md`
  범위의 Phase 2+ 작업이라 아직 미착수다. 즉 위 "잠금 사유 표시"·"비용 상시 노출" UI 개선은
  지금 진행해도 무방하나, **"판독 콘솔처럼 보이지만 판독할 내용이 얕다"는 리뷰의 핵심 지적
  (§11)은 Phase 2+ 착수 전까지는 근본적으로 해소되지 않는다는 점은 여전히 유효하다.**

**기대 효과**
UI 레벨 개선만으로도 노드 하나를 판단하기 위한 탭 횟수가 줄고 방향 감각이 개선된다. 데이터
레벨 개편까지 함께 진행되면 노드 카드가 실제로 "전략적 판단이 필요한 콘솔 항목"으로 기능해,
UI와 게임플레이 양쪽에서 컨셉이 일치하게 된다.

## 7. 수정 우선순위

우선순위는 (a) `UI_UX_Review.md`의 A항목(반드시 수정) 우선, (b) 코드/아트 비용 대비 체감
효과, (c) 다른 항목의 전제 조건 여부 3가지 기준으로 정했다.

| 순위 | 항목 | 근거 |
|---|---|---|
| 1 | 버튼 시스템 재설계(2절) | 전 화면에 반복되는 문제라 한 번 정의하면 파급 효과가 가장 크고, 코드 스코프도 "클래스 추가"로 제한적(8절) — 리스크 대비 효과가 가장 좋음 |
| 2 | ~~HUD Country Dock 2단계화~~ + 정보 중복 정리(5절) | **Country Dock 부분은 컴포넌트 자체가 제거되어 moot(문서 무결성 감사, 2026-07-22, §5 배너 참고)** — 정보 중복 정리(population-bar/graph-panel) 부분만 유효 |
| 3 | 지도 완화책 — 다크 스크림 오버레이(1절 저비용안) | 화면에서 가장 넓은 면적의 문제를 가장 적은 비용으로 즉시 완화할 수 있는 유일한 항목 — 근본책(원본 재도색)보다 먼저 시도 |
| 4 | 국가 선택 화면 accent-bar 강화(4절) | CSS 수준 조정만으로 가능한 부분(국기 크기·채도)을 먼저 처리, 상세 패널 구조 변경은 후순위로 분리 |
| 5 | UpgradeTree UI 레벨 개선(6절 UI 부분) | UpgradeTreeView.cs 한 파일 확장으로 끝나는 범위이나, 데이터 재설계와 맞물려야 완전한 효과가 나므로 4번보다 뒤로 배치 |
| 6 | 병원체 선택 화면 상세패널 개편(3절) | 화면 자체의 문제 심각도가 상대적으로 낮고(리뷰 B항목), 카드 6종 노출 이슈 확인이 선행되어야 함 |
| 7 | 지도 근본책 — world_base.png 재도색 + 교통 아이콘 교체(1절 고비용안) | 효과는 가장 크지만 대형 아트 리소스 작업이라 별도 일정으로 분리 — 3번(완화책) 적용 후 체감 효과를 보고 착수 여부 재판단 권장 |
| 8 | ~~UpgradeTree 데이터 재설계 MVP(6절 데이터 부분)~~ | **[문서 무결성 감사, 2026-07-22] 구현 완료됨** — `UpgradeTree_FinalDecision.md` 기준 `DefaultUpgradeTreeFactory.cs`에 이미 반영 확인. 더 이상 이 계획서의 대기 항목이 아니다. |

1~6은 서로 독립적으로 진행 가능하다(순서를 바꿔도 blocking 관계 없음). 8번(수치 재설계)은
이미 완료됐지만, 6번이 완전히 해소되려면 여전히 `ResearchDatabase_RuntimeSystems.md` 범위의
Phase 2+(노드 효과 다중화)가 필요하다는 점은 유의한다 — §6 참고.

## 8. 예상 개발 비용

중간/높음" 난이도 척도를 그대로 따른다. **코드를 직접 작성하지 않고, 실제 착수 시 건드릴
파일·범위만 근거로 추정한 것**이다.

| 지도 근본책(world_base.png 재도색 + 교통 아이콘 교체) | 높음 | 4000×1714px 배경 원본 재도색(대형 아트 작업) + 국가 실루엣 오버레이 48장과의 색 정합성 재검증 + `TransportUnit.cs`가 로드하는 스프라이트(`plane.png`/`Ship 1.png` 등) 교체(로더 코드 자체는 변경 없음, 애셋 교체만) | 코드 비용은 낮지만 아트 리소스 제작 비용이 이 계획서에서 가장 큼 — 별도 일정 확보 필요 |
| ~~UpgradeTree 데이터 재설계 MVP~~ | 매우 낮음(수치만) | `DefaultUpgradeTreeFactory.cs` 수치 조정 위주, UI 코드 무변경 | **구현 완료됨(문서 무결성 감사, 2026-07-22)** — 이 표에서 더 이상 대기 항목 아님 |

**총평**: 이 계획서의 항목 대부분은 "새 컴포넌트를 발명"하는 것이 아니라 **이미 DESIGN.md에
정의된 패턴(world-status-frame, stat-chip, data-row)을 아직 적용되지 않은 영역으로
확장**하는 작업이라 코드 비용은 전반적으로 낮은 편이다. 예외는 지도 근본책(아트 리소스 규모
문제)과 UpgradeTree 데이터 재설계(밸런스 검증 필요)로, 이 둘만 별도 일정으로 분리해도 나머지
항목은 비교적 짧은 호흡으로 진행할 수 있다.

## 9. DESIGN.md 수정 필요 여부

**결론: 부분 수정 필요.** 이 계획서가 제안하는 항목은 대부분 기존 컴포넌트의 적용 범위
확장이라 새 축을 만들 필요는 없지만, 두 가지는 정본 문서에 없던 개념이라 명문화가 필요하다.

| 수정 대상 | 필요 이유 | 제안 내용 |
|---|---|---|
| Component Library — 버튼 컴포넌트 신설 | 현재 `tab-button`만 정의돼 있고, 화면별로 흩어진 나머지 버튼(주 액션/보조 액션/프리미엄)은 정본 문서에 아예 없다 — "이미 있는데 문서화가 안 됐다"가 아니라 애초에 시스템 밖에 있었다 | `tactical-button`(가칭) 항목 추가 — primary/secondary/premium 3등급, 배경 대신 테두리로 위계 표현, 코너컷 대신 미니 브라켓 사용 규칙을 Usage Rules에 별도 절로 추가 |
| Design Principle #2(Always-dark canvas) — 지도 레이어의 위치 명확화 | 현재 문서는 "모든 화면의 바닥은 다크"라고 선언하지만, 세계지도처럼 화면의 일부만 차지하는 "게임 콘텐츠 레이어"가 이 원칙 안에 포함되는지 예외인지 명시하지 않는다. 이번 리뷰에서 드러난 문제의 근본 원인 중 하나가 바로 이 경계의 불명확함이다 | "지도·게임 콘텐츠 레이어도 다크 캔버스 원칙의 적용 대상"이라는 문장을 명문화하고, 콘텐츠 색(감염 점, 국가 실루엣 등)은 severity/구조색 축을 그대로 따른다는 점을 교차 참조로 추가 |
| Usage Rules — "통제 불가능한 콘텐츠색"(국기·실사 아이콘) 처리 규정 | 국기처럼 게임 콘텐츠상 임의로 색을 바꿀 수 없는 요소가 4축 색상 체계 밖에 존재한다는 점이 지금 문서에는 언급이 없다 — 4번째 절 색상 축 표에 없는 색이 화면에 나타났을 때 "이게 원칙 위반인지 예외인지" 판단 기준이 없었다 | Status Semantics 표 아래에 "콘텐츠 색(국기, 실사 아이콘 등)은 4축에 억지로 편입하지 않되, 크기·채도를 제한해 severity/구조색 신호를 가리지 않아야 한다"는 예외 규정 한 항목 추가 |
| Component Library — 잠금 사유 표시 | 신규 컴포넌트 정의는 불필요 — `stat-chip`을 그대로 재사용하는 것으로 충분하므로 별도 항목 추가 없이 기존 정의의 "용도" 예시에 "노드 잠금 사유 표시"를 한 줄 추가하는 정도로 충분 | Component Library `stat-chip` 항목 설명에 사용례 한 줄 추가(선택 사항) |

나머지(국가 선택 accent-bar 강화 등 — Country Dock 2단계화는 컴포넌트 제거로 moot, 위 §5 참고)는
이미 DESIGN.md에 정의된
`world-status-frame`/Severity Colors 규칙을 그대로 재사용하는 것이라 **문서 수정이 필요
없다** — 적용 범위만 넓히면 된다.
