## 1. 현재 UI의 디자인 언어 분석

스크린샷 4장을 함께 놓고 보면 이 게임은 사실상 **두 개의 서로 다른 디자인 언어가 화면 단위로
번갈아 나타나는 상태**다.

- **언어 A — 전술 콘솔**: UpgradeTree 노드 카드(각진 테두리, 초록 발광 테두리로 "해금 가능"
  표시), HUD 하단 스탯 3분할(감염자/사망자/치료제, severity 색상), 스파크라인 그래프. 이 요소들은
  DESIGN.md가 정의한 "각짐+헤어라인+발광색" 문법을 실제로 따르고 있다.
- **언어 B — 라이트 캐주얼 모바일**: 화면의 물리적 대부분을 차지하는 세계지도(밝은 청록 바다+흰
  대륙), 국기 아이콘이 그대로 노출되는 국가 리스트, 둥근 모서리 카드와 꽉 채운 단색 pill 버튼
  (구매/다음/시작/뒤로/업그레이드/국가현황/랭킹). 이 요소들은 색상·형태 모두 DESIGN.md의 규칙
  (다크 캔버스, 각짐, 헤어라인, severity 4색 고정)과 무관하게 동작한다.

문제는 이 두 언어가 **화면 전체 단위로는 구분되지 않고, 한 화면 안에서 위아래로 뒤섞여 있다**는
점이다. UpgradeTree 화면 안에서도 노드 카드(언어 A)와 "구매" 버튼(언어 B)이 같은 화면 하단
120px 안에 붙어 있고, GamePlay 화면에서도 상단 event-dock/country-dock(언어 A 지향)과 화면
중앙을 채우는 밝은 지도(언어 B)가 동시에 보인다. 플레이어 입장에서는 "이 게임이 진지한 전술
시뮬레이션인지, 캐주얼한 모바일 경영 게임인지"에 대한 신호가 화면마다, 심지어 같은 화면 안에서도
계속 바뀌는 셈이다.

## 2. Tactical Command Center 컨셉과 충돌하는 요소

DESIGN.md가 명시한 5대 시그니처(코너컷/발광 헤어라인/좌측 accent-bar/stat-chip/영문 대문자
캡션)와 Design Principle("항상 다크 캔버스", "각짐")을 기준으로, 스크린샷에서 직접 충돌하는
요소를 뽑으면 다음과 같다.

| 요소 | 위치 | 충돌 내용 |
|---|---|---|
| 밝은 세계지도 배경 | ② GamePlay | Design Principle #2 "Always-dark canvas"(모든 화면의 바닥은 다크, 어떤 화면도 밝은 캔버스로 전환되지 않음)와 정면으로 충돌 — 화면 점유율이 가장 큰 요소가 원칙에서 벗어나 있다 |
| ✈️/🚢 이모지 교통 유닛 | ② GamePlay(CLAUDE.md "교통 유닛 그래픽 — 이모지 기반") | DEFCON/Frostpunk류 군사 HUD 참고라는 정체성 선언과 정면 충돌. 이모지는 메신저·캐주얼 UI의 상징이라 "통제센터"가 아니라 "귀여운 모바일 게임" 신호를 즉시 준다 |
| 둥근 pill 버튼(구매/다음/시작/뒤로/업그레이드 등) | ①②③④ 전 화면 공통 | DESIGN.md Do/Don't "box-shadow, 그라디언트, 둥근 pill 버튼을 도입하지 않는다"와 직접 충돌. 패널·카드는 각지게 바뀌었는데 정작 플레이어가 가장 자주 누르는 CTA(주요 행동 버튼)는 하나도 전환되지 않았다 |
| 국기 원색 팔레트 | ③ CountrySelect | Design Principle #4 "색상 축 분리"(브랜드/severity/카테고리/구조색 4축 고정, 신규 축 추가 금지) 바깥에 있는 사실상의 "5번째 색상 축". 48개국 국기가 전부 다른 원색이라 리스트를 스캔할 때 severity 색상 신호(개발수준 accent-bar)보다 국기색이 시각적으로 더 크게 작용한다 |
| 보라색 광고 버튼("광고 보고 DNA +10") | ① UpgradeTree | 색상 자체(brand-premium 보라)는 축 규칙에 맞지만, 형태가 그라디언트가 도는 둥근 pill이라 Angularity 원칙과 충돌. 화면에서 가장 채도 높고 가장 둥근 단일 요소라 코너컷 카드들 사이에서 이질감이 크다 |

**종합**: 화면을 구성하는 "틀"(패널 테두리·데이터 표시 행)은 전술 언어로 전환됐지만, **배경
(지도)과 행동 유도 요소(버튼)라는, 플레이어의 시선이 실제로 가장 오래 머무는 두 영역**은 전환
대상에서 빠져 있다. 결과적으로 "겉은 통제센터, 알맹이는 캐주얼 게임"처럼 읽힌다.

## 3. UI 계층 구조(Hierarchy) 문제

- **Country Dock의 상시 빈 행 문제**: ② GamePlay 화면 우측 상단 "국가를 선택하세요" 패널은
  선택 전에도 인구/감염률/사망률/의료수준/국경/상태/공항·항구 7개 라벨이 전부 "-" 값으로
  깔려 있다. `world-status-frame`은 DESIGN.md에서 "값이 있을 때만 크롬이 등장"하는 패턴으로
  설계됐는데, Country Dock은 같은 철학을 공유하지 않고 빈 값 상태에서도 패널 전체 높이를
  그대로 점유한다 — 정보가 0인 채로 화면의 상당 부분(세로 20% 안팎)을 차지하는 것은 밀도
  원칙("장식보다 데이터 가독성")에 어긋난다.
- **레이어 겹침/블리드**: ① UpgradeTree 화면 상단 모서리에 배경의 이벤트 로그·국가 패널
  텍스트가 옅게 비쳐 보인다. UpgradeTree가 풀스크린 오버레이라면 아래 레이어가 전혀 보이지
  않아야 자연스러운데, 가장자리에 이전 화면 요소가 겹쳐 보이는 것은 z-order/불투명도 처리가
  화면마다 다르다는 신호다. 플레이어에게는 "지금 이 창이 진짜로 다른 화면 위에 뜬 모달인지,
  같은 화면의 일부인지"가 즉각적으로 명확하지 않다.
- **중복 요약 정보**: UpgradeTree(①)와 GamePlay(②) 모두 "감염 0.0%·사망 0.00%" 같은 비율형
  요약 바와, 바로 아래(또는 근접한 위치)에 "감염자: N / 사망자: N / 치료제: N%" 절대값 요약이
  동시에 떠 있다. 같은 정보를 비율과 절대값 두 형태로 반복 노출하는 것은 계층상 같은 층위에
  중복 배치된 것이라, "이 둘이 다른 정보인가?"를 플레이어가 매번 재확인해야 한다.
- **모달 vs 상시 패널의 구분 약함**: DESIGN.md Surface Hierarchy는 5단계(Canvas/Scrim
  bar/Docked panel/Card/Selected)를 명시하지만, 스크린샷만으로는 event-dock(2단계)과
  country-dock(2단계)이 population-bar/graph-panel(1단계, scrim bar)과 시각적으로 뚜렷하게
  구분되지 않는다 — 배경 알파 계단 차이가 완화 압축된 스크린샷에서는 거의 같은 톤으로 보인다.

## 4. 색상 시스템 문제

DESIGN.md의 4축 분리(브랜드/severity/카테고리/구조색) 자체는 설계로는 명료하지만, 스크린샷에는
그 축 어디에도 속하지 않는 색이 화면을 지배하고 있다.

- **지도 색이 가장 큰 문제다.** GamePlay 화면 면적의 절반 이상을 차지하는 밝은 청록 바다·흰
  대륙은 4축 어디에도 속하지 않는 별도의 색 체계다. HUD 크롬(상하단 바)만 다크+발광색 규칙을
  지키고, 정작 화면에서 가장 넓은 영역(지도)은 전혀 다른 색채 논리로 그려진다. 이 상태에서는
  "다크 캔버스 위에 발광 크롬"이라는 시스템 정체성이 지도 위에서는 성립하지 않는다.
- **국기 색이 severity 신호를 희석한다.** CountrySelect의 accent-bar(개발수준 기반 파랑/회색/
  빨강, §9 설계)는 국기 아이콘의 원색(빨강/주황/파랑/초록 등 국가마다 전부 다름) 옆에 있으면
  상대적으로 존재감이 약해진다. "이 나라가 위험한 발원지인가"라는 전략적 신호보다 "이 나라 국기가
  무슨 색인가"라는 장식적 신호가 먼저 눈에 들어올 가능성이 크다.
- **severity 색상 자체는 잘 지켜지고 있다.** GamePlay 하단 감염자(주황)/사망자(빨강)/치료제
  (청록) 3분할 스탯은 DESIGN.md 5번 표(status-infected/dead/info)와 정확히 일치한다. 이 부분은
  시스템이 실제로 잘 동작하는 사례로 남겨둘 만하다.
- **선택 강조(금색)도 일관적이다.** MainMenu의 병원체 카드, CountrySelect의 국가 행 모두 선택
  시 금색 계열 테두리로 강조되어 `--color-border-selected` 규칙을 따르고 있다 — 규칙이 실제로
  전 화면에 통일되어 있다는 점은 긍정적이다.

## 5. 버튼 디자인 문제

가장 일관되게 반복되는 문제다. **패널/카드는 전부 각진 테두리+헤어라인으로 바뀌었는데, 정작
버튼은 어느 화면에서도 전환되지 않았다.**

- 구매(①)/다음(④)/시작·뒤로(③)/업그레이드·국가현황·랭킹(②) — 전부 배경을 꽉 채운 단색
  둥근 사각형이다. DESIGN.md Component Library의 `tab-button`(주 액션=발광 테두리, 보조
  액션=중립 테두리, **테두리로만 구분**하는 방식)과는 반대로, 실제 버튼은 배경색 채우기 방식
  하나로만 존재한다.
- ①의 "구매" 버튼은 비활성 상태에서 거의 배경과 같은 무채색 회색이라 카드 영역의 발광 테두리와
  나란히 있을 때 대비가 약하다 — "지금 누를 수 있는 행동이 있다"는 신호가 다른 UI 요소보다
  약하게 전달된다.
- 보라색 광고 버튼은 반대로 채도가 가장 높고 그라디언트가 도는 pill이라 과도하게 도드라진다.
  같은 화면 안에서 한쪽(구매 버튼)은 존재감이 약하고 다른 쪽(광고 버튼)은 지나치게 강해,
  "어떤 버튼이 이 화면의 주 행동인가"에 대한 위계가 뒤바뀌어 보일 위험이 있다.
- 버튼만 놓고 보면 4개 화면이 전부 동일한(그리고 유일하게) "일반 모바일 앱" 언어를 공유한다 —
  역설적으로 버튼은 이 게임에서 가장 화면 간 일관성이 높은 요소이지만, 그 일관된 언어가
  DESIGN.md가 정의한 언어가 아니라는 점이 문제다.

**플레이어 체감**: 카드나 배경보다 버튼은 플레이어가 직접 누르는 대상이라 시선이 오래 머문다.
그 순간 게임이 "통제센터"에서 "흔한 캐주얼 앱"으로 되돌아가는 인상을 매번 준다 — 특히 첫
행동(다음/시작)과 마지막 행동(구매) 모두 버튼이라, 화면 진입 시와 이탈 시 두 시점 모두에서
정체성이 흔들리는 셈이다.

## 6. 정보 밀도 및 가독성 분석

- **잠긴 노드가 정보를 주지 않는다.** ① UpgradeTree에서 회색 "상태: 잠김" 카드가 다수 반복된다.
  전부 동일한 텍스트("상태: 잠김")만 보여줄 뿐, 무엇이 부족해서 잠겨 있는지(선행 노드 미해금인지
  DNA 부족인지)는 카드를 눌러 상세 패널을 열기 전까지 알 수 없다. 회색 카드 개수 자체는 많은데
  각각이 주는 정보량은 "잠김"이라는 한 단어뿐이라 밀도 대비 정보 효율이 낮다.
  UI_Design.md §7.1이 이미 "상태가 사실상 이분법이라 LOCKED/AVAILABLE이 구분되지 않는다"고
  자체 진단한 문제와 같은 계열이다.
  주의: DevLog Step 62 기준 4단계 상태 클래스(LOCKED/AVAILABLE/ACTIVE/MAXED) 자체는 구현
  완료로 기록돼 있다 — 스크린샷은 게임 시작 직후(DNA 0) 상태라 ACTIVE/MAXED 노드가 아직
  없어서 2단계만 보이는 것일 수 있다. 다만 "왜 잠겼는지"에 대한 사유는 4단계 색상 구분과는
  별개 정보이며, 카드 표면에는 여전히 노출되지 않는다.
- **Country Dock 유휴 상태의 낭비.** 3절에서 지적한 7행 "-" 표시가 가독성 관점에서도 문제다 —
  실제 정보가 없는데도 시선이 그 영역을 훑게 만들어 정작 지도 위 감염 상황을 보는 데 쓸 여유를
  갉아먹는다.
- **요약 정보 중복(비율 vs 절대값)**도 6절 관점에서 다시 문제가 된다 — 같은 화면에서 "감염
  0.0%"와 "감염자: 108"을 동시에 읽어야 하는 것은 추가 인지 부담이다. 굳이 나누려면 하나는
  상시 노출, 다른 하나는 상세 패널로 옮기는 편이 밀도를 낮춘다.
- **스탯 바(■□) 표기(④)는 반대로 밀도-가독성 균형이 좋은 사례다.** 전염력/중증도/치사율/내성
  4개 축을 5칸 블록으로 압축해 세 병원체를 동시에 스캔·비교할 수 있게 한다 — 데이터 판독
  중심(Readout-first) 원칙에 가장 잘 부합하는 요소 중 하나.

## 7. 병원체 선택 화면(MainMenu) UX 분석

- 3종(세균/바이러스/곰팡이) 카드가 스크롤 없이 한 화면에 다 보이는 점은 초반 의사결정 화면으로
  적합하다 — 비교가 목적인 화면에서 항목을 한눈에 볼 수 있어야 한다는 원칙에 부합한다.
  UI_Design.md §8도 "카드가 6개(실제 화면상 3개 노출)뿐이라 여유 있게 꾸며도 되는 화면"이라고
  미리 짚었는데, 실제 결과도 그 의도에 맞게 동작하고 있다.
  (병원체는 설계상 6종이나 스크린샷에는 3종만 보임 — 스크롤 유무는 이 스크린샷만으로 확인 불가.)
- **이름 중복 노출**: 선택된 "세균" 카드가 위쪽 리스트에 이미 있는데, 아래 detail-panel에서
  "세균"이라는 이름이 다시 헤더로 나온다. 정보량 자체는 적지만, 좁은 세로 화면에서 같은 텍스트가
  두 번 반복되는 것은 detail-panel의 존재 이유(추가 설명)를 다소 흐린다 — 리스트 쪽 선택 카드를
  펼침(expand) 형태로 통합하는 대안도 검토할 만하다.
- **코너컷/발광 테두리의 시각적 존재감이 약하다.** DevLog Step 65 기준 pathogen-card는 이미
  `tactical-panel`+코너컷 4개가 적용된 것으로 기록돼 있으나, 실제 스크린샷에서 카드 간 구분은
  선택 카드의 금색 굵은 테두리가 압도적으로 두드러지고, 미선택 카드의 옅은 회색 테두리·코너컷은
  거의 눈에 띄지 않는다 — "선택 여부"는 잘 보이지만 "이 화면이 통제센터 판독 카드"라는 인상은
  옅다.
- FlavorText(설명 문단)를 문단 그대로 유지한 것은 §8 설계 의도(서술형 텍스트는 data-row로
  쪼개지 않음)와 일치하고, 실제로도 이 정도 텍스트양에서는 문단이 자연스럽다.

## 8. 국가 선택 화면(CountrySelect) UX 분석

- **48개국 스크롤 리스트에서 국기 색이 만드는 시각적 노이즈**는 4절에서 지적한 그대로다.
  실제 의사결정에 필요한 신호(인구 규모, 개발수준=대응력)보다 국기 원색이 먼저 인지되기
  쉬워, "전략적으로 유리한 발원지를 고른다"는 게임 목적보다 "국기가 익숙한 나라를 고른다"는
  감정적 선택으로 흐를 여지가 있다.
- **상세 패널이 화면 하단을 고정 점유**한다 — 선택한 국가(중국)의 인구/기후/의료수준 3줄과
  뒤로/시작 버튼이 하단에 고정되면서, 스크롤 가능한 리스트 영역이 좁아져 한 화면에 5~6개국
  밖에 보이지 않는다. 48개국을 훑어야 하는 화면에서 리스트 가시 영역이 좁은 것은 탐색 비용을
  높인다.
- **개발수준 → severity 색상 매핑(§9 설계)이 실제로 스캔 가능한 수준으로 대비가 나는지는
  스크린샷만으로 확신하기 어렵다** — 국기색·선택 강조색과 섞이면서 옅은 accent-bar 색상까지
  구분해내기가 쉽지 않다. 이 부분은 실기기 검증이 필요한 항목으로 남겨둔다.
- Country Dock(HUD)과 동일한 `data-row` 문법을 상세 패널에 재사용한 것(§9 핵심 요구사항)은
  설계 의도대로라면 "게임 시작 전 미리 보기=게임 중 표시"라는 일관성을 주는 좋은 선택이다 —
  다만 이 일관성은 두 화면을 나란히 비교해야 체감되는 것이라, 단일 스크린샷만으로는 그 효과가
  잘 드러나지 않는다.

  플레이어에게 그리 매력적인 첫 화면은 아니다.

## 10. 업그레이드 시스템(UpgradeTree) UX 분석

- 노드가 초록 발광 테두리(해금 가능)와 회색(잠김)으로 뚜렷이 나뉘어 "지금 무엇을 살 수
  있는지"는 한눈에 파악된다 — 이 부분은 성공적이다.
  선/후행 관계를 연결선(회로도 스타일)으로 보여주는 것도 "무엇이 무엇을 여는지"에 대한 공간적
  직관을 준다.
- 반면 **"왜 잠겼는지"(선행 미해금인지 DNA 부족인지)는 카드 표면에서 구분되지 않는다**(6절).
  또한 카드 표면에 비용(DNA 수치)이 상시 노출되는지, 탭해야만 상세 패널에서 보이는지가
  스크린샷상 일관되지 않게 보인다 — 일부 카드는 이름+상태만 보이고 비용 정보가 카드 밖(상세
  패널)에서만 확인되는 것으로 보인다. §7.3 설계안(코드/이름/상태/비용 4줄 스택)이 실제로는
  이름+상태 2줄 위주로 보여, 노드 하나를 판단하기 위해 매번 탭이 필요할 가능성이 있다.
- 상단 카테고리 전환 화살표(◀▶)로 공기전파/수인성전파/접촉전파 등 갈래를 넘나드는 구조는
  9개 노드씩 카테고리 3개로 나뉜 것과 맞물려 탐색은 되지만, 전체 트리 45개 노드 중 "지금
  보고 있는 게 몇 번째 카테고리인지"에 대한 진행 표시(예: "1/3")가 화면에 없다면 방향 감각을
  잃기 쉽다 — 이 부분도 실기기 확인이 필요하다.
- 하단 "구매" CTA가 5절에서 지적한 대로 카드들의 발광 테두리 대비 존재감이 약해, 노드를
  선택한 뒤 "다음 행동(구매)"으로 이어지는 흐름이 시각적으로 매끄럽지 않다.

## 11. 현재 트리 구조가 게임 컨셉과 맞는지 평가

이 항목은 순수 UI 문제가 아니라 **트리에 들어있는 데이터 구조 자체의 문제**이며, 이미
당시 별도 조사가 코드 레벨로 진단을 마쳐둔 상태였다(조사 문서 자체는 결론이 `UpgradeTree_FinalDecision.md`로
흡수된 뒤 삭제됨). UI 리뷰 관점에서 그 결론을 다시 짚으면:

- 45개 노드 중 약 71%(32개)가 **단일 스탯만 바꾸는 노드**다 — 화면에는 "해금 가능/잠김"
  이분법만 보이지만, 그 이면에는 "이 노드를 사면 명백히 좋기만 하고 대가가 없다"는 구조가
  깔려 있다. UI가 아무리 "전술적 판단 콘솔"처럼 보이게 꾸며져도, 실제 선택지 자체에 트레이드
  오프가 없다면 그 판독 UI는 장식에 가까워진다 — **정보 판독(Readout-first) 원칙이 "판독할
  전략적 갈등이 있을 때"에만 의미를 가진다는 점에서, UI 디자인 문제와 트리 설계 문제가
  서로 얽혀 있다.**
- 카테고리(전파/증상/능력) 간 상호작용이 전혀 없어, "이 카테고리를 밀면 저 카테고리에서
  대가를 치른다"는 긴장이 없다. 상세 패널에 아무리 `data-row`로 효과를 나열해도, 나열된
  값들이 전부 상승(+)뿐이라면 "판단"이 아니라 "확인" 작업이 된다.
- 시기 게이팅(치료제 진행률/발각도 기준 해금 제한)이 없어 DNA만 쌓이면 최단 경로로 최종
  노드까지 직행할 수 있다 — "지금 이 타이밍에 이 노드를 사도 되는가"라는, Tactical Command
  Center 컨셉이 원래 강조해야 할 시간 압박형 의사결정이 트리 구조에는 반영되어 있지 않다.

**결론**: UI 표면(노드 카드, 상태 색상, 상세 패널)은 "전술적 의사결결정 콘솔"처럼 보이도록
잘 다듬어졌지만, 그 밑에 있는 트리 데이터는 아직 "정답이 하나뿐인 스탯 상승 목록"에 가깝다.
컨셉과 UI/데이터 사이의 이 간극은 이미 프로젝트 내부적으로 인지되어 별도 조사 문서로 정리돼
있으므로, 이번 리뷰에서는 "UI 리뷰만으로는 해결되지 않는 문제"라는 점을 재확인하는 선에서
정리한다.

| 교통/유닛 그래픽 | 단순 아이콘(비행기/배 실루엣), 군더더기 없는 절제된 스타일 | 이모지(✈️) 사용 | 원작보다 톤이 가볍다 — "통제센터"를 표방하면서 원작보다 캐주얼한 아이콘을 쓰는 것은 정체성상 역행 |
| 색상 절제 | 감염 단계 색 + UI 크롬 색 정도로 제한적 팔레트 | severity 4색+브랜드 3색 체계는 설계상 원작 수준으로 절제돼 있으나, 국기 색이라는 통제되지 않은 팔레트가 추가로 섞여 있음(4절) | 설계는 원작 수준이나 실제 화면(국가 리스트)에서는 원작보다 색이 더 많다 |
| 특성 카드 시각 위계(잠금/해금) | 초반부터 이유(비용 부족/전제조건 미충족)가 잠금 아이콘 위에 함께 표시되는 경우가 많음 | "상태: 잠김" 텍스트만 반복(6·10절) | 원작이 "왜 안 되는지"를 더 즉각적으로 전달한다 |

**요약**: 시뮬레이션 데이터 구조(국가 붕괴 단계 등)는 이미 원작 수준을 따라잡았지만, **UI가
전달하는 "긴장감 있는 절제된 톤"은 오히려 원작보다 못한 지점(밝은 지도, 이모지 아이콘, 잠금
사유 미표시)이 있다.** "Plague Inc보다 더 군사적/전술적인 느낌을 주겠다"는 이 프로젝트의
지향점을 기준으로 보면, 현재 상태는 그 목표에 원작보다 가까워졌다기보다 오히려 일부 지점에서
더 멀어져 있다.

## C. 유지해야 하는 항목

| 항목 | 유지 이유 |
|---|---|
| HUD 하단 severity 색상 체계(감염=주황/사망=빨강/치료제=청록) | DESIGN.md 5번 표와 정확히 일치하며 스크린샷에서도 일관되게 지켜짐(4절) — 시스템이 실제로 잘 동작하는 사례 |

| `data-row`(라벨:값) 패턴 자체 | Country Dock·CountrySelect 상세 패널·UpgradeTree 상세 패널이 같은 문법을 공유하도록 설계된 것은 "같은 통제센터의 다른 콘솔"이라는 인상을 만드는 핵심 장치(9·10절에서 이미 그 의도가 확인됨) — 표면 디테일은 손봐도 이 패턴 자체는 유지할 가치가 있음 |
