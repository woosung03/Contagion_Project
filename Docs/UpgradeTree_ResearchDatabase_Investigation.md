# UpgradeTree 재설계 조사 — "노드+연결선 트리" vs "연구 데이터베이스"

조사 전용 문서. **코드/UXML/USS 작성 없음.** `UI_UX_Review.md`/`UI_Redesign_Plan.md`(검토 완료)와
`UpgradeTree_Redesign_Investigation.md`/`UpgradeTree_MVP_ChangeProposal.md`/
`UpgradeTree_FinalDecision.md`(노드 데이터 재설계, 반영 확정)는 이미 다른 질문에 답했다 — "트리 안의
노드 효과가 얼마나 얕은가"(데이터 문제)와 "트리 UI를 어떻게 다듬을까"(잠금 사유·비용 표시·진행
인디케이터 등, 표면 문제)였다. 이번 조사는 그 두 문서가 다루지 않은 **더 상위의 질문** —
"노드+연결선 캔버스라는 시각적 틀 자체를 유지할지, 리스트 기반 연구 데이터베이스로 바꿀지" — 를
다룬다. 반영 여부·범위는 다음 세션에서 결정.

---

## 0. 결론 먼저 — 이 조사가 기존 3개 문서와 충돌하지 않는 이유

`UpgradeTree_FinalDecision.md`가 확정한 MVP(노드 `cost`/`effects` 수치 교체, 표시명 1개 교체)는
**데이터 레이어**만 건드리고 `UpgradeTreeView.cs`(UI 코드)는 전혀 손대지 않는다. 이번 조사가 다루는
"트리 캔버스 → 리스트" 전환은 **프레젠테이션 레이어**만 건드리고 `UpgradeNode`/`UpgradeManager`/
`DefaultUpgradeTreeFactory`(데이터 모델)는 전혀 손대지 않는다. 두 작업은 서로 다른 레이어라 순서
상관없이 독립적으로 진행 가능 — 이미 결정된 데이터 MVP를 재검토하거나 되돌릴 필요가 없다.

## 1. 현재 UpgradeTree가 Tactical Command Center 컨셉과 얼마나 충돌하는가

**전제부터 하나 교정할 필요가 있다.** "노드를 연결선으로 잇는 트리 구조 자체 = RPG"라는 전제는
정확하지 않다 — Plague Inc. 원작도 DNA 화면이 노드 그리드 + 연결선 구조다(12절 비교표 참고).
장르 자체가 이미 이 문법을 쓴다. 그런데도 원작은 "연구소"처럼 느껴지고 우리 프로젝트는 "스킬
트리"처럼 느껴진다면, 문제는 트리라는 **틀(container)**이 아니라 그 틀 안에 담긴 내용과 지금
이 프로젝트에서 그 틀이 놓인 맥락일 가능성이 크다. 구체적으로 세 가지가 겹쳐서 RPG 인상을 만든다.

1. **노드 효과 깊이 (이미 별도 문서가 진단·처방함)** — 45개 중 71%가 단일 스탯만 바꾼다
   (`UpgradeTree_Redesign_Investigation.md` 0절). "이 노드는 순수 버프, 저 노드도 순수 버프"가
   반복되면 트리 모양과 무관하게 "스탯 찍기"로 읽힌다. 이건 `FinalDecision.md` MVP가 이미
   해소 대상으로 확정한 문제라 이번 조사에서 다시 다루지 않는다.
2. **명명 규칙이 스탯 그라인딩을 연상시킨다** — "공기 전파 I/II/III", "구조 강화 I/II" 같은 로마
   숫자 연쇄 티어링은 MMO/RPG 스킬트리의 표준 명명법과 거의 동일하다. Plague Inc. 원작은 같은
   전파경로 안에서도 "공기 전파 I/II/III"가 아니라 이름 자체가 갈리는 특성(예: Bird 1 → 감염력
   축, Rodent 1 → 별도 감염력 축)을 쓰거나, 이름이 갈려도 효과가 뒤섞여 "레벨업"처럼 안 느껴지게
   한다. 이 부분은 트리냐 리스트냐와 무관하게 **어느 쪽을 택해도 남는 문제**다 — 6절에서 별도로
   다룬다.
3. **이 화면만 프로젝트 전체에서 유일하게 "공간적 캔버스" 렌더링을 쓴다** — Country Dock,
   CountryPopup, CountryStatusPanel, CountrySelect, MainMenu, RankingPanel(예정)까지 프로젝트의
   나머지 모든 화면은 예외 없이 "리스트 + 상세 패널"(`data-row` 판독 행) 문법을 쓴다(`DESIGN.md`
   Surface Hierarchy, 판독 우선 원칙). UpgradeTree만 절대좌표 배치(`node.position.x/y`) +
   Painter2D로 그린 꺾은선 연결선을 쓴다(`UpgradeTreeView.cs` `RebuildTree()`/`DrawConnections()`).
   **"같은 통제센터의 다른 콘솔"이라는 인상을 만드는 게 이 프로젝트 UI 설계의 핵심 장치인데
   (`UI_UX_Review.md` 9·10절), UpgradeTree만 그 문법 밖에 있다.** 이게 "RPG스럽다"는 인상의
   상당 부분을 설명한다 — 노드가 몇 개 카드로 보이든, 다른 5개 화면과 렌더링 방식이 다르다는
   사실 자체가 이 화면을 "이질적인 화면"(그리고 게임 UI에서 이질적인 화면은 관성적으로 "스킬
   트리 메뉴"로 읽힌다)으로 만든다.

**결론**: 컨셉 충돌은 실재하지만, 원인은 "트리 모양"보다 "노드 얕음(이미 처방됨) + 명명 규칙 +
이 화면만 다른 렌더링 문법을 쓴다는 것" 세 가지의 합이다. 세 번째 원인은 트리를 리스트로 바꾸면
자동으로 해소되고, 첫 번째는 이미 해소 경로가 있으며, 두 번째는 트리를 유지하든 리스트로
바꾸든 별도로 손봐야 한다.

## 2. 현재 트리 구조가 플레이어에게 주는 인상

`UI_UX_Review.md` 10절이 이미 스크린샷 기준으로 진단했다 — 잠김/해금 가능 이분법은 잘 읽히고
연결선이 "무엇이 무엇을 여는지"의 공간적 직관은 준다는 점은 성공적이라고 평가됐다. 다만:

- 노드 카드가 코드/이름/상태/비용 4줄 스택 + 각진 테두리 + 발광 헤어라인이라는 이 프로젝트
  고유의 "판독" 톤을 입고는 있지만, **그 카드들을 절대좌표 캔버스 위에 흩뿌리고 선으로 잇는
  전체 레이아웃 자체**는 스킬트리 장르의 표준 레이아웃과 형태적으로 동일하다. 표면 스타일(색/
  폰트/테두리)은 Tactical이어도 **레이아웃 문법**은 RPG 트리 그대로라, 플레이어가 무의식적으로
  참조하는 "이 화면과 닮은 다른 게임 화면"은 결국 스킬트리 메뉴다.
- 카테고리 전환이 좌우 화살표로 되어 있어 "지금 3개 중 몇 번째를 보는지" 감각이 약하다(이미
  `UI_Redesign_Plan.md` 6절이 "1/3 인디케이터 추가"를 처방했지만, 이는 트리 캔버스 구조를
  유지한 채 부족한 정보를 덧붙이는 패치다).
- 노드 45개, 카테고리당 9개라는 규모 자체는 크지 않은데도 캔버스 방식은 "전체를 한눈에 조망"이
  가능해야 그 이점이 산다 — 지금은 카테고리당 9개만 그리도록 이미 쪼개놨고(원래 27개를 한
  캔버스에 넣었다가 방향 감각 상실 피드백으로 분리한 이력, 코드 주석 12~23행), 그마저도 참조
  해상도 480px 폭 안에 다 넣으려고 열 간격을 180→140으로 줄인 상태다. **캔버스가 주는 이점
  (공간적 조망)을 이미 스스로 포기해가며 화면에 욱여넣고 있다** — 이는 캔버스 방식이 이 세로
  모바일 화면 크기와 원래 잘 안 맞는다는 신호로 읽을 수 있다.

## 3. RPG 스킬트리 vs Research Database 비교

| 항목 | 현재(노드+연결선 캔버스) | Research Database(탭→목록→상세) |
|---|---|---|
| 의존 관계 표현 | 연결선으로 공간적·시각적 직관 제공 (성공적이라는 평가 있음, 2절) | 텍스트로 표현("선행 연구: OOO 완료") — 공간적 직관은 없지만 지금 상세 패널이 이미 이렇게 하고 있어(11.4절 코드) 학습 비용 없음 |
| 화면 폭 적합성 | 절대좌표라 좁은 세로 화면에 맞추려고 열 간격을 이미 축소한 상태 — 확장 여지가 좁음 | 세로 스크롤 리스트는 폭 제약이 원천적으로 없음, 모바일 세로 화면의 기본 문법 |
| 이 프로젝트 다른 화면과의 일관성 | 유일하게 캔버스+Painter2D 렌더링 — 5개 화면(Country Dock/Popup/StatusPanel/Select/MainMenu)과 문법이 다름(1절) | `data-row`/리스트 문법 재사용 — 나머지 전 화면과 동일한 "판독 콘솔" 인상 |
| 장르 관성 | Plague Inc·스킬트리류 모두가 쓰는 익숙한 형태 (원작과 형태적으로 가장 가까움) | 인벤토리/카탈로그/연구소 UI에 가까움 — "RPG 성장"보다 "정보 열람·선택" 인상이 강함 |
| 잠금 사유 표시 | 카드가 좁아 "상태: 잠김" 텍스트만 반복(이미 지적된 문제) | 리스트 행 폭이 넓어 "잠김 · 선행: 기침 필요" 한 줄에 바로 표시 가능 — 기존 문제 해결과 전환이 동시에 됨 |
| 카테고리 진행 감각 | 좌우 화살표 + 별도 "1/3" 인디케이터 추가가 필요(처방됨) | 탭 3개를 상단에 항상 노출하면 "지금 어디 있는지"가 인디케이터 없이 항상 보임 |
| 코드 복잡도 | 절대좌표 계산(`_xOffset`, `canvasWidth/Height`) + Painter2D 꺾은선 벡터 계산 유지 | 해당 계산 로직 전체 제거, `ScrollView` 기본 레이아웃으로 대체(8절) |
| 구현 리스크 | 낮음(이미 동작 중) | 낮음(데이터 계층 무변경, 상세 패널 100% 재사용 — 8·9절) |

## 4. Research Database 전환 시 장단점

**장점**

- 1·3절에서 짚었듯 이 화면만 이질적이던 렌더링 문법이 사라져 "같은 통제센터의 다른 콘솔"이라는
  전체 인상이 완성된다 — 지금 유일하게 비어 있는 마지막 퍼즐 조각.
- 잠금 사유 미표시·비용 표시 불일치·카테고리 진행 감각 부재라는, `UI_UX_Review.md`가 이미
  "반드시/하면 좋음" 항목으로 지목해둔 문제 3개가 전환 과정에서 부수적으로 해결된다(3절 표).
- 좁은 세로 화면에 절대좌표를 욱여넣던 제약이 사라져 노드(연구 항목) 설명에 여유 공간이 생긴다
  — `sym_multiorgan1` 같은 복잡한 다중 효과 노드일수록 리스트 행이 카드보다 정보를 더 넉넉히
  보여줄 수 있다.
- 코드량이 순감소한다(8절) — 유지보수 표면이 줄어든다.

**단점**

- 연결선이 주던 "이 노드 두 개가 이 노드 하나로 합류한다"는 공간적 직관이 사라진다. 특히 지금
  `trans_advanced1`/`trans_advanced2` → `trans_global` 처럼 두 갈래가 하나로 합류하는 지점은
  선이 없으면 텍스트만으로는 "합류"라는 느낌이 약해질 수 있다. 리스트에서는 "선행 연구: 광역
  전파 I, 광역 전파 II" 두 줄로 표현되긴 하지만, 한눈에 안 들어온다.
  → 완화책: 합류 노드는 리스트 상에서 "선행 2개 완료 시 해금"처럼 별도 배지로 강조하거나,
  해당 갈래 섹션 맨 위에 "이 갈래의 최종 목표"로 고정 배치하는 정도로 보완 가능(구현 시 논의).
- Plague Inc. 원작과의 형태적 유사성이 줄어든다 — 원작 팬에게 익숙한 "그 DNA 화면" 느낌이
  약해질 수 있다. 다만 이 프로젝트의 지향점 자체가 "원작보다 더 군사적/전술적인 느낌"
  (`UI_UX_Review.md` 12절 요약)이므로, 원작과 달라지는 것 자체가 반드시 손해는 아니다.
- GDD §7.2 와이어프레임("노드 트리 스크롤 뷰")이 트리를 전제하고 있어, 이 문서도 함께
  갱신해야 한다(경미한 문서 정합성 작업, 반영 확정 시 처리).

## 5. 선행조건 시스템 유지 방안

선행조건(`prerequisites: List<string>`) 데이터 모델은 **손댈 필요가 전혀 없다.** 이미 상세
패널(`SelectNode()`)이 `AddDetailRow("선행 노드", string.Join(", ", ...))`로 텍스트 표시를
하고 있어(11.4절 코드), Research Database 전환은 이 표현 방식을 "탭 후 상세 패널에서만"이
아니라 "리스트 행 자체에서도" 보이게 확장하는 정도다.

구체적 표현 방안(설계 수준, 구현 아님):

- **잠긴 행에 사유를 인라인 표시** — "잠김 · 선행: 기침" 처럼 상태 캡션 옆에 붙인다. 이건
  `UI_UX_Review.md` B항목("잠긴 노드의 사유 미표시")이 이미 처방한 것과 동일한 해법인데,
  트리 카드보다 리스트 행이 가로 폭이 넓어 자연스럽게 들어간다.
- **갈래(strand)를 섹션 헤더로 유지** — "은신형 / 표준형 / 공격형"(FinalDecision.md가 이미
  Symptom에 도입 확정한 갈래 명칭) 같은 소제목으로 리스트를 묶으면, 티어 순서(1→2→3)는
  섹션 안의 위쪽에서 아래쪽 순서로만 표현해도 충분하다 — 연결선 없이도 "이게 이 갈래의 몇 번째
  단계"라는 감각은 유지된다.
- **합류/최종 노드는 시각적으로 구분** — 두 갈래 이상을 선행으로 요구하는 노드(`trans_global`,
  `superbug1` 등)는 리스트에서 별도 배지("통합 연구")나 섹션 최하단 고정 배치로 다른 노드와
  구분한다.

## 6. 전파/증상/능력 카테고리 구조 유지 여부

**유지를 권장한다.** 이유는 세 가지다.

1. 이 구조는 시각적 틀(트리 vs 리스트)과 독립적인 **데이터 분류 축**이다 — `UpgradeCategory`
   enum이 이미 이 3분류로 고정돼 있고, `UpgradeTree_FinalDecision.md`가 확정한 데이터 재설계도
   이 3분류를 그대로 쓴다(1절 "Transmission/Symptoms/Resistance 역할 재정의" — 카테고리 수는
   그대로 두고 노드 효과만 재설계).
2. Research Database 구조(탭 선택 → 목록)와 정확히 대응된다 — 사용자가 예시로 든 [TRANSMISSION]
   [SYMPTOMS] [ABILITIES] 탭이 곧 지금의 3카테고리다. 전환에 새 분류 체계가 필요 없다.
3. `UpgradeTree_FinalDecision.md`가 이미 카테고리 내부에 **갈래(strand)**라는 2단계 분류를
   도입 확정했다(은신형/표준형/공격형, 약물내성/백신회피/면역교란) — 이건 사용자가 제안한
   "탭 → 연구 항목 목록"의 "목록" 부분을 이미 한 단계 더 세분화해둔 것과 같다. 즉 데이터
   레이어가 이미 "탭(카테고리) → 그룹(갈래) → 항목(노드)" 3단 구조로 굳어지는 중이라, 이번에
   제안하는 UI 구조가 데이터 구조와 자연스럽게 들어맞는다.

## 7. UI/UX 관점에서 가장 적합한 구조 제안

3단 구조를 권장한다 — 사용자가 제시한 방향과 일치하되, 이 프로젝트에 이미 있는 컴포넌트로
구체화한다.

```
[상단 탭 3개] 전파 / 증상 / 능력  (카테고리, 항상 노출 — 좌우 화살표+진행 인디케이터 대신
                                    탭 자체가 위치를 상시 표시)
   ↓
[섹션 리스트] 갈래(strand)별 헤더 + 그 아래 세로 리스트
   예) "은신형" 섹션 — 기침 / 만성피로 / 발진 / 피부병변 / ...
       "표준형" 섹션 — 발열 / 폐렴 / ...
   각 행 = code + 이름 + 상태칩(잠김/가능/진행중/완료) + 비용(또는 "완료") + (잠겨있으면
   선행 사유 인라인)  — 기존 tree-node의 4줄 정보를 한 줄~두 줄 리스트 행으로 압축
   ↓
[상세 패널] 탭 시 하단(또는 별도 패널)에 기존 SelectNode()가 이미 그리는 그대로 —
   코드/명칭/효과별 data-row/선행 노드/DNA 비용/상태/구매 버튼. 변경 없음.
```

이 구조가 적합한 근거: (a) 다른 5개 화면과 동일한 "리스트+상세 패널" 판독 문법을 그대로
따르므로 프로젝트 전체의 시각적 일관성이 완성된다(1절), (b) 세로 모바일 화면에 스크롤 하나로
자연스럽게 들어맞아 현재 캔버스가 겪는 폭 제약이 사라진다(2·4절), (c) 선행조건·잠금사유·비용
표시라는 기존 UI 문제 3개가 별도 작업 없이 구조 전환 과정에서 함께 풀린다(4절).

## 8. 구현 난이도 비교

`UpgradeTree_Redesign_Investigation.md` 6절이 쓰는 척도(매우 낮음/낮음/중간/높음)를 그대로 쓴다.

| 방향 | 난이도 | 근거 |
|---|---|---|
| 현행 유지 + UI 폴리싱만(잠금사유/비용통일/진행표시/구매버튼 승격) | 낮음~중간 | 이미 `UI_Redesign_Plan.md` 6·7절이 산정 완료 — `CreateNodeElement()` 확장 정도, 순위 5번으로 이미 스코프됨 |
| **Research Database 전환** | **낮음** | `RebuildTree()`의 절대좌표 계산(`_xOffset`, `canvasWidth/Height` 산출)과 `DrawConnections()`의 Painter2D 벡터 계산(elbow 좌표·포트 마커, 약 60줄)을 통째로 제거하고, `CreateNodeElement()`를 리스트 행 생성 함수로 교체(box.style.position=Absolute 대신 `ScrollView` 기본 흐름 사용)하면 된다. **제거되는 코드가 새로 필요한 코드보다 많다** — 순수 UI 레이어 교체이며 게임 로직·데이터 모델은 1줄도 안 건드린다(9절) |

두 방향 모두 낮은 난이도지만, Research Database 쪽이 (a) 코드가 순감소하고 (b) UI 폴리싱이
해결하려던 문제들을 부산물로 해결하며 (c) 향후 유지보수 표면이 더 작다는 점에서 실질적으로
더 유리하다.

## 9. 기존 시스템 재사용 가능 범위

재사용률이 매우 높다 — **데이터 계층은 100% 무변경, UI 로직도 대부분 재사용된다.**

| 구성요소 | 재사용 여부 |
|---|---|
| `UpgradeNode`/`UpgradeManager`/`DefaultUpgradeTreeFactory`(데이터 모델·해금 로직) | 100% 무변경 |
| `NodeDisplayNames`(45개 표시명 딕셔너리) | 100% 그대로 |
| `DetermineState()`/`StateCaption()`(잠김/가능/진행중/완료 판정) | 100% 그대로 — 리스트 행 상태칩에 그대로 씀 |
| `SelectNode()`/`AddDetailRow()`(상세 패널, "연구 분석 콘솔") | 100% 그대로 — 이미 리스트형 상세 패널이라 트리든 리스트든 무관하게 동작 |
| `GetEffectiveCost()`/`CanUnlock()`/`EffectStatLabel()` | 100% 그대로 |
| `CategoryLabel()`/`CategoryPrefix()`/`CategoryCaption()`(탭 라벨) | 100% 그대로 — 탭 UI로 그대로 전용 가능 |
| `OnPrevRequested`/`OnNextRequested` + 3-GameObject 카테고리 구조 | 그대로 유지 가능(탭 전환으로 의미만 재해석), 또는 탭 바 UI로 대체 시 소폭 수정 |
| `stat-chip`/`data-row`(DESIGN.md 공용 컴포넌트) | 다른 5개 화면에서 이미 검증된 컴포넌트를 그대로 재사용 |
| `RebuildTree()`의 절대좌표 계산, `DrawConnections()`(Painter2D 연결선) | **제거 대상** — 유일하게 버려지는 부분 |

즉 버려지는 코드는 "노드를 캔버스 위에 어디에 그릴지 계산하는 부분"뿐이고, "무엇을 보여줄지"를
결정하는 로직(상태 판정·비용 계산·표시명·상세 정보 조립)은 전부 그대로 살아남는다. 이는 원래
`UpgradeTreeView.cs`가 데이터 기반 렌더링으로 설계돼 있었기 때문(`Redesign_Investigation.md`
6절이 이미 "노드 효과값·비용·표시명을 바꾸는 것만으로는 UI 로직을 전혀 안 건드린다"고 확인한
것과 같은 맥락 — 이번엔 반대로 "UI 로직(레이아웃 방식)을 바꿔도 데이터·상세 패널 로직은 안
건드린다"는 대칭적 결론).

## 10. Contagion Project 정체성에 가장 적합한 최종안

**Research Database 방향으로 전환을 권장한다.** 근거를 요약하면:

- 1절에서 밝혔듯 "트리=RPG"라는 전제 자체는 정확하지 않지만(원작도 트리다), **이 프로젝트
  한정으로는** 트리 캔버스가 나머지 5개 화면과 문법이 다른 유일한 화면이라는 점이 RPG 인상의
  실질적 원인이다 — 리스트 전환은 이 원인을 구조적으로 제거한다.
- 노드 효과 얕음(가장 큰 원인)은 이미 `UpgradeTree_FinalDecision.md`가 별도 레이어에서
  해결 중이므로, 이번 결정은 그 작업과 배타적이지 않다. 오히려 두 작업이 합쳐지면 "데이터도
  깊어지고 표현도 이 프로젝트 고유의 판독 문법으로 통일되는" 이중 효과를 낸다.
- 카테고리(전파/증상/능력)·갈래(은신형/표준형/공격형 등)·선행조건 시스템 전부 그대로 유지되므로
  "정답 트리 없는 전략적 선택"이라는 게임플레이 목표에는 영향이 없다 — 이번 결정은 순수하게
  "그 선택지를 어떤 화면 문법으로 보여줄 것인가"의 문제다.
- 구현 난이도·재사용률(8·9절) 모두 현행 유지보다 불리하지 않다 — 오히려 코드가 줄고 기존
  UI 버그 3개가 부산물로 해결된다.

**단, 두 가지는 별도로 명시해둔다.**

1. 명명 규칙(로마 숫자 I/II/III 연쇄)은 트리든 리스트든 남는 문제다(1절 2번) — 이번 결정과
   별개로, 노드 이름 자체를 "단계형"에서 "정체성형"으로 바꾸는 작업(예: "공기 전파 II" →
   "에어로졸 확산")을 함께 검토할 가치가 있다. 이번 조사 범위 밖이라 별도 트래킹만 남긴다.
2. 시기 게이팅 부재(`UpgradeTree_Redesign_Investigation.md` 2단계 항목, 아직 미반영)는 트리든
   리스트든 동일하게 남아 있는 "DNA만 있으면 최단 경로 직행 가능" 문제다 — 화면을 리스트로
   바꿔도 이 문제는 저절로 해결되지 않는다. 계속 별도 2단계 항목으로 트래킹.
